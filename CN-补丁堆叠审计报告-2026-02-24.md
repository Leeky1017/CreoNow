# CN 代码库「补丁堆叠」审计报告

> 审计日期：2026-02-24
> 审计范围：`apps/desktop/main/src`、`apps/desktop/renderer/src`、`packages/shared/`

## 总览

经过对全量源码的系统性分析，识别出 **7 类结构性补丁堆叠模式**，共涉及约 **45 个独立问题实例**。

代码库**未出现** `@ts-ignore`/`@ts-expect-error`、`as any`（生产代码零匹配），类型纪律良好。问题集中在架构层的重复定义、遗留兼容适配层、和掩盖根因的降级链上。

---

## 类别一：基础设施类型/函数大规模复制粘贴

**严重程度：高** | **实例数：~30 个文件** | **性质：本应抽取为共享模块，却在每个服务文件中独立定义**

### 1.1 `ipcError()` 函数重复定义（30 个文件）

完全相同的 `ipcError(code, message, details?)` 工厂函数在以下 30 个文件中各自独立定义：

- `services/kg/kgCoreService.ts`

- `services/context/contextFs.ts`
- `ipc/constraints.ts`
- `services/export/exportService.ts`
- `services/search/searchReplaceService.ts`
- `services/search/hybridRankingService.ts`
- `services/search/ftsService.ts`
- `services/context/synopsisStore.ts`
- `services/embedding/embeddingService.ts`
- `services/skills/skillExecutor.ts`
- `services/ai/aiProxySettingsService.ts`
- `services/embedding/semanticChunkIndexService.ts`
- `services/memory/memoryTraceService.ts`
- `services/memory/preferenceLearning.ts`
- `services/rag/ragService.ts`
- `services/ai/providerResolver.ts`
- `services/memory/userMemoryVec.ts`
- `services/skills/skillLoader.ts`
- `ipc/stats.ts`
- `services/ai/judgeQualityService.ts`
- `services/context/watchService.ts`
- `services/ai/traceStore.ts`
- `services/stats/statsService.ts`
- `services/judge/judgeService.ts`
- `services/skills/skillScheduler.ts`

**根因**：最初某个服务文件写了 `ipcError()` 后，后续服务通过复制粘贴方式引入，从未抽取为共享模块。

**正确做法**：在 `main/src/ipc/` 或 `main/src/services/shared/` 下建一个 `ipcResult.ts`，导出 `ipcError()`、`Ok<T>`、`Err`、`ServiceResult<T>`，所有服务统一 import。

### 1.2 `ServiceResult<T>` / `Ok<T>` / `Err` 类型重复定义（25+ 个文件）

与 `ipcError()` 完全相同的分布。每个服务文件独立声明：

```ts
type Ok<T> = { ok: true; data: T };
type Err = { ok: false; error: IpcError };
export type ServiceResult<T> = Ok<T> | Err;
```

部分文件还有细微变体（如 `services/documents/types.ts` 的 `DocumentError` 替代 `IpcError`），增加了不一致风险。

### 1.3 `nowTs()` 函数重复定义（11 个文件）

完全相同的 `function nowTs(): number { return Date.now(); }` 在以下 11 个文件中独立定义：

- `services/projects/projectService.ts`
- `services/ai/aiProxySettingsService.ts`
- `services/documents/documentCoreService.ts`
- `services/skills/skillService.ts`
- `ipc/pushBackpressure.ts`
- `ipc/ai.ts`
- `logging/logger.ts`
- `services/memory/preferenceLearning.ts`
- `services/memory/memoryTraceService.ts`
- `services/memory/userMemoryVec.ts`
- `services/memory/memoryService.ts`

**影响**：如果未来需要统一注入 fake timer（P6 确定性原则要求），需要修改 11 个位置而非 1 个。

## 类别二：同步/异步函数完整复制（非重构为单一实现）

**严重程度：中高** | **实例数：4 对函数** | **性质：代码逻辑完整复制而非共享核心逻辑**

### 涉及文件：`services/context/contextFs.ts`

| 同步版本                       | 异步版本                            | 行号          |
| ------------------------------ | ----------------------------------- | ------------- |
| `ensureCreonowDirStructure()`  | `ensureCreonowDirStructureAsync()`  | 39-85 / 345-395 |
| `getCreonowDirStatus()`       | `getCreonowDirStatusAsync()`        | 92-109 / 400-417 |

两对函数的业务逻辑（目录列表、默认文件列表、错误处理）完全一致，仅 `fs.mkdirSync` ↔ `fsPromises.mkdir` 等 API 不同。任何业务变更都必须同步修改两处。

**根因**：最初只有同步版本，后来为了避免阻塞主进程事件循环又添加了异步版本，但没有重构原有同步版本。

**正确做法**：保留异步版本作为 SSOT，移除同步版本或让同步版本调用异步版本（如果确实需要同步路径）。

## 类别三：遗留兼容适配层（Legacy Adapter Shims）

**严重程度：中** | **实例数：5 处** | **性质：旧 API 签名保留，通过适配器转发到新实现，而非直接迁移调用方**

### 3.1 KG Store 的 `toLegacyEntity()` / `toLegacyRelation()` 适配层

**文件**：`renderer/src/stores/kgStore.ts:138-154`

```ts
function toLegacyEntity(entity: EntityResponse): KgEntity {
  return { ...entity, entityId: entity.id, entityType: entity.type, metadataJson: toMetadataJson(entity.attributes) };
}
function toLegacyRelation(relation: RelationResponse): KgRelation {
  return { ...relation, relationId: relation.id, fromEntityId: relation.sourceEntityId, toEntityId: relation.targetEntityId };
}
```

IPC 契约已使用 `id`/`type`/`sourceEntityId`/`targetEntityId`，但 renderer 组件仍依赖 `entityId`/`entityType`/`fromEntityId`/`toEntityId`。适配层在 `createKgStore` 的 6 个方法中反复调用（行 197-198, 318, 372, 416, 442），而非直接让组件使用新字段名。

**正确做法**：统一 renderer 组件使用 IPC 原生字段名，删除 `KgEntity`/`KgRelation` 扩展类型和 `toLegacy*` 函数。

### 3.2 KG Store 的 `normalizeEntityType()` 中的 `"other" → "faction"` 映射

**文件**：`renderer/src/stores/kgStore.ts:101-122`

```ts
case "other":
  return "faction";
```

这是一个硬编码的遗留映射：旧版本的 UI 用了 `"other"` 类型，但后端只接受 `"faction"`。正确做法是修正 UI 层不再产生 `"other"` 值。

### 3.3 Memory Store 的 `bootstrapForProject()` 向后兼容包装

**文件**：`renderer/src/stores/memoryStore.ts:343-346`

```ts
/** @deprecated Use bootstrapForContext instead */
bootstrapForProject: async (projectId) => {
  // Backward compatible wrapper
  await get().bootstrapForContext(projectId, null);
},
```

标记为 `@deprecated` 但仍保留。应当找到所有调用方并迁移，然后删除此方法。

### 3.4 AI Proxy Settings 的遗留字段兼容

**文件**：`services/ai/aiProxySettingsService.ts:279-300`

`getRaw()` 返回值中保留了 `baseUrl`/`apiKey` 两个遗留字段（`normalizedLegacyBaseUrl`/`normalizedLegacyApiKey`），同时这些值还被用作新字段 `openAiCompatibleBaseUrl`/`openAiByokBaseUrl` 的 fallback：

```ts
openAiCompatibleBaseUrl: normalizeBaseUrl(openAiCompatibleBaseUrl) ?? normalizedLegacyBaseUrl,
openAiByokBaseUrl: normalizeBaseUrl(openAiByokBaseUrl) ?? normalizedLegacyBaseUrl,
```

`update()` 方法中也保留了 `encryptedLegacyKey` 的处理路径（行 449-493）。

### 3.5 Provider Resolver 的多层 fallback 链

**文件**：`services/ai/providerResolver.ts:81-145`

`resolveSettingsProviderCredentials()` 中每个 provider 的 credentials 都有三级 fallback：

```ts
const openAiCompatible: ProviderCredentials = {
  baseUrl: args.settings.openAiCompatible?.baseUrl
    ?? args.settings.openAiCompatibleBaseUrl
    ?? args.settings.baseUrl
    ?? null,
};
```

这三级是因为 `ProxySettings` 类型同时保留了旧的扁平字段（`openAiCompatibleBaseUrl`）和新的嵌套字段（`openAiCompatible.baseUrl`），再加上最古老的 `baseUrl`。`resolveSettingsBackupProvider()` 在行 185-220 又完整重复了一遍这些 fallback 构造。

**正确做法**：统一 `ProxySettings` 的数据结构为单一形式，在读取层做一次归一化，消除下游的多级 fallback。

## 类别四：`as unknown as` 类型强转掩盖契约不一致

**严重程度：中** | **实例数：~10 处生产代码** | **性质：类型不匹配时用强转绕过编译器，而非修正底层类型定义**

### 涉及文件

- `renderer/src/features/settings/AiSettingsSection.tsx` — 类似模式
- `renderer/src/features/outline/deriveOutline.ts` — 3 处 `as unknown` 强转

这些强转说明 IPC 响应类型与 renderer 组件期望的类型存在不一致。正确做法是统一类型定义，消除强转需要。

## 类别五：`eslint-disable` 掩盖的函数规模问题

**严重程度：低中** | **实例数：2 处** | **性质：用 eslint-disable 绕过代码质量规则而非重构**

### 涉及文件

- `renderer/src/stores/editorStore.tsx:128` — `// eslint-disable-next-line max-lines-per-function`
- `preload/src/ipcGateway.ts` — 1 处 eslint-disable

`editorStore.tsx` 的 `createEditorStore` 工厂函数超长，用 eslint-disable 抑制了 `max-lines-per-function` 规则。正确做法是将内部逻辑拆分为独立函数（如 `createSaveQueue()`、`createBootstrapActions()` 等）。

## 类别六：降级链掩盖根因

**严重程度：中** | **实例数：3 处** | **性质：通过 graceful degradation 掩盖了应该修复的底层问题**

### 6.1 Rules Fetcher 的双重降级

**文件**：`services/context/fetchers/rulesFetcher.ts`

当 `entityList` 返回 `!ok` 时返回空 chunks + warning；外层 catch 又做了同样的降级。如果 KG 服务长期不可用，上下文层会静默忽略所有规则注入，没有任何告警上报机制。

### 6.2 Embedding Service 的多层 fallback

**文件**：`services/embedding/embeddingService.ts:295-344`

`encode()` 中当 `modelId === "default"` 时有：primary provider → classify failure → check fallback policy → fallback provider → final error。这个链本身设计合理，但 `classifyPrimaryFailureReason` + `shouldFallback` 的逻辑过度复杂，且 fallback 失败后静默丢弃错误只返回 primary 的错误信息。

### 6.3 Memory Injection Preview 的 semantic → deterministic 降级

**文件**：`services/memory/memoryService.ts:880-909`

当 `vecRes` 失败时，默默回退到 deterministic 模式，返回所有记忆项。诊断信息埋在 `diagnostics.degradedFrom` 中，但调用方没有对此做任何处理。

const candidates = uniquePaths([

## 非问题（已验证的健康指标）

- **零 `@ts-ignore` / `@ts-expect-error`** — 类型纪律严格
- **零 `as any`**（生产代码）— 类型安全完整
- **IPC 运行时校验层** (`runtime-validation.ts`) 设计正确，是单一边界守卫
- **全局异常处理** (`globalExceptionHandlers.ts`) 设计合理，无补丁
- **DB 初始化失败** 允许 `db: null` 传播是有意设计，不是补丁
- **文档服务**已有 `document-service-no-duplicate-implementation.test.ts` 守卫，防止遗留 SQL 回归

## 修复优先级建议

1. **P0**：抽取 `ipcError()` + `ServiceResult<T>` + `nowTs()` 到共享模块（影响 30+ 文件，单次重构即可完成）
2. **P1**：消除 `contextFs.ts` 的同步/异步双份代码
3. **P1**：统一 `ProxySettings` 数据结构，消除三级 fallback
4. **P2**：迁移 KG Store 调用方到 IPC 原生字段名，删除 `toLegacy*`
5. **P2**：修复 `as unknown as` 强转点的底层类型不一致
6. **P3**：删除 `bootstrapForProject()` deprecated 方法
7. **P3**：拆分 `editorStore.tsx` 的超长工厂函数
