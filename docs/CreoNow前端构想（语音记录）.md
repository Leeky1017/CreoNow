# CreoNow 前端构想（语音记录）

## 说明

这是一份通过语音输入逐步整理的前端设计构想文档。Leeky 用自然语言描述想法，Joy 负责补足技术细节、结构化整理并持续更新。

---

## 前端整体愿景

### 设计哲学：极简但不简陋

- **核心原则：Less is More，复杂留给工程，简单留给用户。**
- 视觉上追求简约、大气、克制——能不加装饰就不加，拒绝花里胡哨。
- 参考标杆：**Cursor**（开发者工具的极简典范）和 **Notion**（信息密度高但视觉干净）。
- 好设计的标准：用户一眼看清「这是什么」，一秒理解「怎么用」。
- 简单感的背后是工程深度——把难的东西消化在底层，呈现层只留最清晰的表达。

> 「简单的前提是，工程师已经把所有非常难的东西都做好了，但呈现出来的东西给人一眼就能看清楚。」 —— Leeky
> 

### Agent Native：智能无处不在，但不叫「AI」

- CN 是一个 **Agent 原生产品**，不是「传统产品 + AI 功能」的拼凑。
- 智能不是某个独立模块或按钮，而是渗透在每一个交互环节中——**但这不代表没有独立的对话入口**，恰恰相反，要给用户**多种方式**进入与 Agent 的对话和指令交互。
- 整个前端体验的设计起点是：**用户与 Agent 协作的流畅度**，而非传统的 CRUD 界面。

**命名约束：产品界面中禁止出现「AI」字样**

这是 CN 的一条核心设计约束。参考 Notion 和 Cursor：它们从不会把功能命名为「AI 编辑 xxx」或「AI 助手」，智能能力是融入产品的自然部分，而不是一个需要贴标签的卖点。

- 用 **Agent** — 体现自主性、能动性、个性化
- 不用 **AI** / **LLM** / **模型** — 过于技术化、变成卖点而非体验
- 实操：按钮不叫「让 AI 改写」，而是「润色」「扩写」「翻译」；面板不叫「AI 助手」，而是直接叫 Agent 或赋予一个人格化名称。

这两个原则在前端落地时意味着：

1. **UI 层**：大量使用留白、一致的 Design Token、极少的视觉层级（参考 Notion 的 3 级灰度体系）。
2. **交互层**：减少显式操作步骤，多用 inline 提示、上下文感知菜单、渐进式展开（Progressive Disclosure）。
3. **Agent 层**：智能能力以 ambient（环境式）方式存在——自动补全、智能建议、上下文理解——但不只是一个独立的对话框，而是通过多通道入口（Inline、Side Panel、Full Panel）让用户随时进入协作。
4. **工程层**：需要强健的抽象（ProseMirror Schema/Plugin 体系 + Zustand 状态管理 + IPC 契约），让底层复杂度不泄漏到 UI。

## 核心模块与布局

整体采用 IDE 风格的布局，参考 VS Code 的空间结构，但服务于文字创作场景。

### 布局总览

```jsx
┌───┬────────┬────────────────────────────────┬───────────────┐
│   │        │  Top Bar（工具栏 + 文件切换）    │               │
│   │        ├────────────────────────────────┤               │
│ I │  Left  │                                │    Right      │
│ c │  Side  │      Main Editor               │    Side       │
│ o │  Panel │      （主编辑区）               │    Panel      │
│ n │ （可    │                                │   （Agent     │
│   │  收起） │                                │    Panel）    │
│ B │        │                                │               │
│ a │        │                                │               │
│ r │        │                                │               │
├───┴────────┴────────────────────────────────┴───────────────┤
│  Status Bar（状态栏）                                            │
└────────────────────────────────────────────────────────────┘
```

**关键布局决策：Top Bar 属于 Main Editor 内部**

Top Bar 不是横跨整个窗口的全局栏，而是**嵌在 Main Editor 区域的顶部**。这意味着：

- Icon Bar、Left Side Panel、Right Side Panel 都是从顶到底的全高区域
- Top Bar 只占 Main Editor 的宽度，不侵占左右面板的空间
- 视觉上，左右面板和 Icon Bar 是「外壳结构」，Main Editor + Top Bar 是「内容区域」
- 好处：当用户收起左右面板进入沉浸写作时，Top Bar 自然跟随编辑区，不会留下一条突兀的全宽顶栏

### 1. Icon Bar（图标栏 / 活动栏）

最左侧一条窄窄的竖向栏，只放小图标，作为各功能区域的入口。类似 VS Code 最左侧的 Activity Bar。

- 每个图标对应一个功能模块（如：文件树、搜索、世界观、设置等）
- 点击图标后的行为有两种模式：
    - **展开左侧边栏**：如文件树、世界观管理——点击后在 Icon Bar 右侧展开一个 Left Side Panel（可收起）
    - **弹出中央面板**：如搜索——直接在屏幕正中央弹出一个 Command Palette / 搜索面板（参考 CN V1 的搜索设计）

**术语解释**

- **Icon Bar / Activity Bar**（图标栏 / 活动栏）：最左侧的窄竖条，只放图标，不放文字。VS Code 里就是那个放着文件、搜索、Git 等图标的最窄的一条。
- **Side Panel**（侧边栏）：图标栏旁边展开的内容区域，宽度比图标栏大得多，里面放具体内容（文件列表、世界观树等）。可以随时收起。
- **Command Palette**（命令面板）：屏幕中央弹出的搜索/命令框。VS Code 里按 Cmd+Shift+P 弹出的那个框。CN 的搜索面板就是这个概念，可以搜到用户能触达的几乎所有内容。

### 2. Left Side Panel（左侧边栏）

点击 Icon Bar 中的某些图标后展开，显示对应功能的详细内容：

- **文件树 / 项目结构**：展示当前项目的所有文档、章节、素材等
- **世界观管理**：角色、地点、设定、时间线等结构化内容
- **其他功能面板**：大纲、版本历史、书签等
- 可随时收起，收起后主编辑区自动扩展填满空间

### 3. Main Editor（主编辑区）

屏幕正中央的核心区域，占据最大面积，用于文字创作：

- 基于 ProseMirror 的富文本编辑器
- 沉浸式写作体验，可收起左右两侧面板获得最大写作空间
- 支持选中文本后的 Floating Toolbar + Inline Agent（已在「交互与体验」章节描述）

### 4. Top Bar（顶部栏）

位于 Main Editor 内部的顶部，包含两个核心区域：

- **工具栏（Toolbar）**：放置常用操作按钮（如保存、导出、设置等）
- **文件切换 Tab**：不采用传统的横向多 Tab 排列（避免打开多个文件后需要拖动滚动条才能找到后面的文件）

#### 文件切换的特殊设计

这是 CN 与传统 IDE 的一个重要差异点：

- 顶部始终只显示一个「当前文件」的 Tab，就是用户正在编辑的那一个文件
- 点击该 Tab → 弹出**文件切换列表（File Switcher List）**
- 列表中每一行显示：文件名称、上次编辑时间、以及其他必要信息
- 用户通过列表点击切换文件，而不是在横向 Tab 条里左右拖拽查找
- 好处：无论打开多少文件，切换体验始终干净、可控，不会出现 Tab 拥挤问题

**术语解释**

- **Toolbar**（工具栏）：一行横向的按钮栏，放着常用操作的快捷按钮。
- **Tab**（标签页）：在这里指顶部显示当前文件名的一个小标签，点击它弹出切换列表。
- **File Switcher**（文件切换器）：点击 Tab 后弹出的列表面板，类似手机上切换 App 的任务列表，但这里切换的是文件。

### 5. Right Side Panel（右侧边栏 / Agent Panel）

右侧主要用于 Agent 面板，详见「交互与体验」章节的「Agent Panel 三种形态」。三种形态：

- **Side Panel 模式**：FAB 单击展开，右侧边栏，边写边聊
- **Full Panel 模式**：FAB 双击展开，近乎全屏，深度对话
- **Mini Panel 模式**：屏幕中央弹出的小面板，最快速触达，适合一句话指令

**术语解释**

- **Workbench Shell**（工作台外壳）：这是一个前端架构术语，指的是以上所有区域（Icon Bar + Left Panel + Main Editor + Right Panel + Top Bar + Status Bar）组合在一起的「外层容器」。就像蛋壳包裹着蛋白和蛋黄一样，Shell 是最外层的「壳」，它负责管理各个区域的位置、大小、显示/隐藏状态。用户看不到它，但它决定了整个界面的空间分配逻辑。简单说：**Workbench Shell = 整个应用的布局框架**。

### 6. Status Bar（状态栏）

屏幕最底部的一条窄横栏，始终可见，显示当前工作状态的关键信息：

- **当前文档字数**：实时显示当前文档的总字数（对写作者来说这是最常看的数据）
- **光标所在章节**：显示当前光标位置对应的最近标题名称，帮助用户感知自己在文档中的位置
- **保存状态**：显示「已保存」或「未保存」，让用户随时知道内容是否安全
- **当前项目模式**：显示当前项目的类型（小说/文案/论文/通用）

**术语解释**

- **Status Bar**（状态栏）：应用窗口最底部的一条窄栏，显示各种实时状态信息。VS Code 底部那条蓝色/紫色的栏就是 Status Bar（显示 Git 分支、编程语言、行号等）。CN 的 Status Bar 显示的是写作相关的状态：字数、章节、保存状态、项目模式。

### 7. Full Editor View（全编辑区视图）

除了 Side Panel 和 Command Palette 两种呈现形态之外，CN 还有第三种重要形态：**工具直接接管主编辑区**。

当用户点击 Icon Bar 中的 Knowledge Graph 或 Characters（角色管理）等功能时，这些工具不应该挤在一个侧边栏或弹窗里——它们需要大面积的可视化空间。此时该工具会**完全替换主编辑区的内容**，成为当前 Tab 的显示内容，就像 VS Code 中打开 Settings 界面或 Git Graph 一样。

这是一个非常合理的设计模式。原因：

- Knowledge Graph 需要大面积画布来渲染节点和关系连线
- Characters（角色管理）需要卡片式/列表式的可视化界面
- 这些本质上是给 Agent 使用的数据，但**人类需要掌控感和可视化**——看到它、理解它、才能信任 Agent 的行为
- Side Panel 太窄装不下，弹窗又不够沉浸，全编辑区才是正确的容器

**术语解释**

- **Full Editor View / Editor Tab**（全编辑区视图 / 编辑器标签页）：一种让非文档内容（如设置界面、可视化工具）占据主编辑区域的模式。用户可以像切换文件一样切换回正在编辑的文档。VS Code 的 Settings 页面、Git Graph 扩展都是这个模式。

### 功能呈现形态总览

每个功能模块可以支持一种或多种呈现形态，下表总结了各功能与其适用的 UI 容器：

| 功能模块 | Left Side Panel | Command Palette | Full Editor View | Right Panel (Agent) | Mini Panel |

## 大纲系统（Outline System）

CN 的大纲系统分为两套，服务于完全不同的场景：

### 1. 创作大纲（Creative Outline）— Full Editor View

这是 CN 的重型大纲工具，为创作者提供完整的构思空间。类比：就像软件开发前写 Spec（规格说明书）一样，用户可以把大纲写得足够详细，详细到 Agent 拿到后能直接生成完整的草稿。

**核心理念：用户可以选择深度**

- 深度模式：用户把大纲完善到 Spec 级别（卷 → 章 → 场景 → 核心冲突 → 角色动机），Agent 根据 Spec 生成完整草稿
- 轻量模式：用户只写一个粗略的思路，然后边写边优化，让 Agent 边写边补足没想到的部分
- 用户可以随时在两种模式之间切换——先写粗略版，逐步细化

**大纲工具集（Outline Toolkit）**

创作大纲不是纯文字列表，它需要多种辅助工具让用户把思路建立得「立体」，而不是只有平面的几行字：

- **结构化编辑器**：卷/章/场景的层级结构，可拖拽排序、折叠展开
- **场景卡片（Scene Cards）**：每个场景是一张卡片，可以包含 POV 角色、地点、时间、核心冲突、伏笔等元数据
- **情节线可视化（Plot Lines）**：主线、副线、角色弧线的可视化展示，让用户看到整个故事的发展脉络
- **Agent 辅助构思**：用户写了一个粗略想法，Agent 可以提问、建议、补充细节，引导用户把大纲做深做完整。就像一个经验丰富的编辑跟作者对谈
- **不同内容类型的大纲模板（Outline Templates）**：针对不同创作场景提供引导性结构（见下方「项目类型与模式」章节）

**大纲 ↔ 正文双向同步（Bidirectional Sync）**

这是创作大纲最核心的能力：

- **大纲 → 正文**：Agent 根据大纲生成正文草稿。大纲越详细，生成质量越高
- **正文 → 大纲**：无论正文是 Agent 生成的、用户自己写的、还是 Agent 优化后的，Agent 都能自动提取和更新大纲，保持大纲与实际内容同步
- 双向同步是持续的：用户修改了正文，大纲自动更新；用户修改了大纲，Agent 提示是否需要同步到正文

**呈现位置**

创作大纲是项目级功能，主要在 Full Editor View 中呈现（因为它需要大量工具和可视化空间），也可以在 Left Side Panel 中显示简略版（只看标题层级）。

**术语解释**

- **Creative Outline**（创作大纲）：CN 的重型大纲工具，让用户拆解、组织、可视化整个作品的结构。不是简单的列表，而是一套完整的构思环境。
- **Spec**（规格说明书）：软件开发中，开发前写的详细设计文档。写得够详细的话，开发者可以直接照着写代码。CN 的创作大纲也是同样的思路——写得够详细，Agent 能直接生成高质量草稿。
- **Scene Cards**（场景卡片）：每个场景的结构化信息卡，类似影视制作中的场景板（storyboard）。
- **Bidirectional Sync**（双向同步）：两个东西互相影响、自动保持一致。改了一边，另一边跟着变。

### 2. 导航大纲（Navigation Outline / TOC）— 页面内置

这是一个轻量级的页面内导航工具，复刻 Notion 右侧内置的 Table of Contents 体验，完全不需要 Agent 参与，自动从文档标题层级提取生成。

**参考原型：Notion 的页面内 Outline**

Notion 在页面右侧有一个内置的 Outline 面板。它的核心交互是：

- 平时（光标不在 Outline 区域时）：只显示一组**灰色短横线**，每条横线代表一个标题。横线的长度和缩进暗示标题的层级（H1 最长、H2 稍短并缩进、H3 更短更缩进）。用户一眼就能感知「这里有个导航，文档大概这么长、这么分层」，但不占注意力
- 当鼠标悬停到 Outline 区域时：横线**动画展开为完整的标题文字**，用户可以看到每个标题的具体内容，点击即可跳转
- 这个设计的精妙之处在于：**不用时极度克制（几条灰线），用时立刻可用（完整标题）**，完美符合 CN 的「Less is More」哲学

**CN 的导航大纲设计**

**位置与布局**

- 固定在 Main Editor 的**右侧边缘**，紧贴编辑区内容，不占用 Right Side Panel（Agent Panel）的空间
- 它是编辑器自身的附属 UI，不是一个独立面板。类比：VS Code 右侧的 minimap（缩略图），它是编辑器的一部分而不是一个独立面板
- 当 Right Side Panel 展开时，导航大纲**不隐藏**，而是跟随 Main Editor 一起被挤压收缩。Right Panel 占的空间越大，Main Editor 越窄，导航大纲占用的屏幕空间也越小——但导航大纲内部的设计比例不变（横线长度关系、字体大小、缩进层级都保持原样）。本质上，导航大纲是 Main Editor 容器的内部子元素，跟随容器宽度自然响应式收缩，不需要任何特殊隐藏/显示逻辑
- 垂直方向上，导航大纲的高度等于编辑器可视区域的高度，不超出

**两种视觉状态**

*状态 A：收起态（Collapsed）— 默认状态*

- 显示为一组细横线，纵向排列
- 每条横线 = 一个 Heading 节点
- 横线的**长度**编码层级：H1 最长（比如 24px）、H2 中等（16px）、H3 最短（10px）
- 横线的**左缩进**也编码层级：H1 无缩进、H2 缩进 8px、H3 缩进 16px
- 横线之间的**垂直间距**反映实际文档中标题之间的相对距离——如果两个标题之间隔了很多正文，对应的两条横线之间间距也更大。这让用户从横线的疏密就能感知文档的内容分布
- 颜色：所有横线使用极淡的灰色（如 `rgba(0,0,0,0.15)` / 暗色主题下 `rgba(255,255,255,0.15)`），不干扰写作
- **当前位置指示器**：用户光标所在位置对应的那条横线高亮为主题强调色（亮色主题下为黑色，暗色主题下为白色），让用户知道「我现在在文档的哪个部分」

*状态 B：展开态（Expanded）— 悬停时*

- 当鼠标进入导航大纲区域时，横线平滑动画展开为标题文字
- 展开动画：横线从左向右「生长」为文字，持续约 150-200ms，用 ease-out 缓动
- 每个标题项显示：缩进层级 + 标题文字（截断过长的标题，末尾用省略号）
- 当前所在标题**加粗**显示
- 标题文字可点击——点击后编辑器平滑滚动到对应标题位置
- 鼠标离开区域后，反向动画收回为横线状态

**交互细节**

- **点击跳转**：点击任一标题，编辑器使用 `scrollIntoView` 将对应标题滚动到视口顶部附近（留一定 offset 避免标题紧贴顶边）
- **双向滚动同步**：用户在编辑器中正常滚动时，导航大纲的「当前位置指示器」实时跟随移动。实现方式：监听编辑器的 scroll 事件，用 Intersection Observer 检测哪个 Heading 当前在视口中，更新高亮状态
- **键盘快捷键**：可以绑定一个快捷键（如 Cmd+Shift+O / Ctrl+Shift+O）直接唤出一个 Quick Outline 弹窗（类似 VS Code 的 Go to Symbol），在弹窗中搜索标题并跳转。这是导航大纲的「键盘友好」补充
- **平滑过渡**：所有状态切换（收起↔展开、当前位置更新）都使用 CSS transition 动画，不允许突兀的跳变

**数据来源与自动更新**

- 从 ProseMirror 编辑器的文档模型（Doc Node）中遍历提取所有 Heading 节点（H1-H6）
- 构建一个 `OutlineItem[]` 数组：`{ level: number, text: string, pos: number }[]`，其中 `pos` 是标题在文档中的位置偏移量
- 监听 ProseMirror 的 `tr`（Transaction）事件，每次文档变更时重新提取标题列表。由于只提取 Heading 节点（通常几十个），性能开销极低
- 注意：完全不需要 Agent 参与，这是纯前端的 DOM + 编辑器状态驱动

**实现方案（给工程师）**

- 创建一个 `<NavigationOutline />` React 组件，挂载在 Main Editor 容器的右侧
- 使用 ProseMirror 的 `editor.state.doc.descendants()` 遍历文档节点，筛选 `node.type.name === 'heading'` 的节点
- 用 React 局部 state 管理 outline items + 当前高亮项（组件内部状态，不需要跨组件共享）
- 收起态用 `<div>` + CSS width/margin 模拟横线；展开态用 `<span>` 渲染文字
- 展开/收起动画用 CSS `max-width` + `opacity` transition
- 滚动同步用 `IntersectionObserver` 监测每个 heading 元素的可见性
- 整个组件大约 200-300 行代码，不依赖任何外部库（除了编辑器本身）

**与创作大纲的区别**

- 导航大纲是**自动的、只读的、轻量的**——它只是帮你快速定位，不涉及内容创作。它永远是一个「附属 UI」，紧贴编辑器
- 创作大纲是**手动的、可编辑的、重型的**——它是作品的「设计图」，用来规划和组织整个创作。它是一个独立的 Full Editor View
- 两者可以同时存在：用户在 Full Editor View 编辑创作大纲时，导航大纲照常工作，帮用户在大纲文档内快速定位

**术语解释**

- **TOC / Table of Contents**（目录）：自动从文档标题生成的导航列表。点击即可跳转到对应位置。Notion 右侧那个就是 CN 要复刻的原型。
- **Heading**（标题）：Markdown 中用 # ## ### 表示的不同级别的标题。导航大纲就是自动收集这些标题生成的可视化列表。
- **Intersection Observer**（交叉观察器）：浏览器原生 API，用来检测一个元素是否进入/离开可视区域。CN 用它来实时追踪「用户现在看到的是哪个标题」。
- **minimap**（缩略图）：VS Code 右侧那个显示代码缩略全貌的窄条。CN 的导航大纲在收起态时的视觉逻辑与 minimap 类似——用极简的视觉元素表达文档全貌。

## 交互与体验

### Agent 入口设计：多通道、渐进式

CN 提供至少三种不同层级的 Agent 交互入口，覆盖从轻量操作到深度对话的全场景：

#### 1. FAB 悬浮球（固定右下角）

参考 Notion 的 FAB 设计，始终固定在右下角，是全局 Agent 入口：

- **单击** → 右侧滑出 Agent 侧面板（Side Panel），适合快速提问、查看建议、执行简单指令。
- **双击** → 展开为近乎全屏的 Agent 对话面板（Full Panel），适合深度对话、长篇协作、复杂任务编排。
- FAB 球本身也可承载状态提示（如 Agent 正在思考时的动画反馈）。

#### 2. 选中文本浮动工具栏（Floating Toolbar + Inline Agent）

CN 作为文字创作工具，编辑器内的选中交互至关重要：

- 用户在编辑器中**选中一段文字**后，弹出浮动工具栏（Floating Toolbar），包含：
    - **常规格式工具**：加粗、下划线、文本颜色、链接、删除等
    - **技能快捷入口**：润色、翻译、扩写、缩写等一键操作（注意：按钮命名直接用动作词，不加「AI」前缀）
- 同时支持 **Inline Agent 对话**：选中文本后，可在工具栏区域内直接输入自然语言指令，Agent 就地处理并替换/编辑选中内容，无需离开编辑上下文。

#### 3. 其他入口

- **快捷键唤起**：见「平台策略与快捷键」章节
- **右键上下文菜单**：见「右键上下文菜单」章节（已有完整设计）

#### 4. Agent Panel 三种形态（Side / Full / Mini）

FAB 悬浮球触发的 Agent Panel 有三种独立形态，覆盖不同深度的交互需求：

- **Side Panel 模式**（FAB 单击）：右侧边栏形态，宽度约 360-400px，编辑器主体左移让出空间。适合边写边聊——查看 Agent 建议、执行简单指令、浏览 Diff 预览。对话状态在 Side Panel 和 Full Panel 之间共享（同一个 Zustand store），切换时上下文连续不丢失
- **Full Panel 模式**（FAB 双击）：近乎全屏的对话界面，编辑器退到背景。适合深度对话、长篇协作、多轮推理、复杂任务编排。可以在对话中引用文档片段、查看完整的 Diff 对比、管理多个候选版本
- **Mini Panel 模式**（快捷键唤起，如 Ctrl+Shift+A）：屏幕中央弹出一个小型浮层（约 480px × 320px），输入一句话指令后自动关闭。适合最快速的 Agent 触达——不需要打开侧边栏，不需要离开当前视野。输入框支持 @ 引用（如 @角色名、@章节名），Agent 执行完毕后 Mini Panel 自动收起，结果直接作用于编辑器

**三种形态的切换关系**

- Side Panel ↔ Full Panel：点击 Side Panel 顶部的「放大」按钮切换到 Full Panel；Full Panel 顶部的「缩小」按钮回到 Side Panel
- Mini Panel 是独立入口，关闭后不影响 Side/Full Panel 的状态
- 三者共享同一个对话历史（Zustand store），用户在任何形态中的对话都保存在同一处

**术语解释**

- **Side Panel**（侧面板）：从屏幕右侧滑出的面板，与编辑器并排显示。宽度可调，但不超过屏幕宽度的 40%。
- **Full Panel**（全面板）：近乎全屏的 Agent 对话界面，适合需要大量上下文的深度交互。
- **Mini Panel**（迷你面板）：屏幕中央的小型浮层，类似 Spotlight / Command Palette 的交互模式，输入完即关闭。

这套多通道设计的核心思路是**渐进式深度（Progressive Depth）**：

- **轻量操作**（选中 → Inline Agent）：不打断写作心流，0 跳转
- **中度交互**（FAB 单击 → Side Panel）：可以边写边看 Agent 输出
- **深度对话**（FAB 双击 → Full Panel）：适合长对话、多轮推理、任务编排

技术上需要注意：

- Side Panel 和 Full Panel 共享同一个对话状态（Zustand store），切换时保持上下文连续。
- Floating Toolbar 的 Agent 区域需要做好与编辑器 Selection API 的联动，确保选区不会因为点击工具栏而丢失。
- Inline Agent 的输出需要支持 diff 预览（类似 Cursor 的绿色/红色标记），让用户决定接受或拒绝修改。

## Agent 集成与协作

### 核心定位

Agent 在 CN 中几乎是「无所不能」的——CN 内置的所有工具，Agent 都可以调用。这是 Agent Native 的真正含义：不是给 Agent 划一块专属区域，而是让它成为整个工作环境的原生参与者。

CN 专注于文本创作（未来可能扩展到图像等），目标是用所有可用工具帮助创作者把文章、小说、论文、以及所有可以用文字表达的东西做到极致。而且要比其他通用 Agent 做得更好，因为 CN 是垂直场景。

### Knowledge Base（知识库）— 统一底层，不同视角

CN 的 Knowledge Graph、Memory、Characters、Worldbuilding 本质上是**同一个系统的不同视角**，统称为 Knowledge Base（知识库）。

**架构关系**

- **Knowledge Graph**（知识图谱）是底层数据结构——所有实体（角色、地点、事件、设定、物品等）及其关联都存储在这里
- **Characters**（角色）是 Knowledge Graph 的一个「滤镜」——只看角色类型的实体和它们的关系
- **Worldbuilding**（世界观）是另一个「滤镜」——看地点、时代、规则、科技等设定类实体
- **Memory**（记忆）是横切面的信息——包括用户偏好、项目上下文、历史对话、以及从 KG 中提取的摘要信息

**用户体验：不同入口，同一底层**

用户不需要知道底层是同一个系统。他们看到的是独立的入口：

- 想管理角色？点击 Characters 入口，看到角色卡片和关系图
- 想管理世界观？点击 Worldbuilding 入口，看到设定树和规则列表
- 想看全局关系？点击 Knowledge Graph 入口，看到完整的节点和连线图
- 想查看/编辑 Agent 的记忆？点击 Memory 入口

底层数据是联通的——在 Characters 里给角色添加了一个关系，Knowledge Graph 里自动出现这条连线。

**Memory 的双向性**

- **自动生成**：Agent 自动记住用户的写作偏好、项目上下文、历史对话，跨会话保持连续性
- **用户可编辑**：Memory 生成后不是被定死的。用户可以手动查看、修改、删除任何记忆条目。比如用户可以告诉 Agent：「记住：这个项目里‘老王’是反派，但读者到第 10 章之前不应该知道」
- **CN 最终服务的是用户，不是 Agent**。用户对 Memory 有完全的控制权

**Agent 的其他内置能力**

- **SKILLS（技能）**：内置的文本操作能力（润色、扩写、缩写、翻译、风格转换、一致性检查等），以及全球 Agent 生态中已有的通用能力
- 未来可扩展：图像生成、语音输入等

**术语解释**

- **Knowledge Base**（知识库）：KG + Memory + Characters + Worldbuilding 的统称。底层是同一个数据系统，上层通过不同视角展示给用户。
- **Knowledge Graph**（知识图谱）：「节点 + 连线」的数据结构。角色 A「认识」角色 B，地点 X「位于」城市 Y。是 Knowledge Base 的底层引擎。
- **Memory**（记忆）：Agent 的持久化上下文。自动生成 + 用户可编辑，双向的。

### 用户自定义

用户可以自定义 Agent 的：

- **技能组合**：选择启用哪些 SKILLS，添加自定义 SKILLS
- **人格**：设定 Agent 与用户对话时的语气和态度（注意：人格 ≠ 写作风格，见「写作风格与语气控制」章节）
- **上下文权限**：决定 Agent 可以访问哪些项目数据

### Agent 输出的呈现方式

Agent 生成内容后，提供多种方式让用户审阅和采纳：

- **Diff 对比（Diff View）**：类似代码编辑器的红绿标记，显示 Agent 修改前后的差异，用户可以逐处接受或拒绝。
- **候选列表（Candidates / 「抽卡」模式）**：一次性生成多个版本，用户浏览并挑选最满意的一个。不是为了消耗 token 而多生成，而是为创作者提供选择的自由度。
- **直接替换**：对于简单操作（如翻译），可以直接替换原文，用户可 Undo 撤回。

**术语解释**

- **Diff View**（差异视图）：显示修改前后对比的界面。红色 = 删除的内容，绿色 = 新增的内容。类似 Git diff 或 Cursor 的 AI 编辑预览。
- **Candidates**（候选项）：同一个指令生成多个不同的结果，让用户挑选。就是你说的「抽卡」模式。

## 写作风格与语气控制（Writing Style & Tone）

用户可能全程使用 Agent 写作，因此风格控制是 CN 的核心能力。

### 风格的两个层级

- **项目级（Project Level）**：整个项目统一的写作风格。比如一本小说的整体语调、人称、目标读者等。Agent 生成的所有内容都遵循这套设定。
- **段落级（Paragraph Level）**：同一篇文章内可以切换风格。比如小说的叙事部分用文艺风，对话部分用口语风，内心独白用意识流风格。

### 风格 = SKILL（技能）

风格模板本质上就是 Agent 的 SKILL 系统的一部分：

- CN 可以**预设一些基础风格 SKILL**（如「严肃文学风」「网文爽文风」「商业文案风」「学术论文风」），作为起点
- 但真正好的 SKILL **不是别人给的，而是用户自己打磨的**。CN 提供思路和起点，用户根据自己的需求持续优化，变成最适合自己的 SKILL
- **「像某个作家的风格」**也可以被固化成一个 SKILL——比如用户创建一个「东野圭吾风格」 SKILL，Agent 就模仿那种写法
- 用户也可以让 Agent 自己探索风格——不需要预设模板，而是通过对话让 Agent 理解用户想要的感觉

### 风格 ≠ Agent 人格（Style ≠ Personality）

这两个概念必须分开设置：

- **风格（Style）**：指的是**文章本身**的写作语调、用词习惯、句式结构。影响的是 Agent 生成的正文内容。
- **人格（Personality）**：指的是 **Agent 与用户对话时**的语气和态度。比如活泼/沉稳/毒舌/温柔。影响的是 Agent Panel 里的对话体验。
- 举例：Agent 的人格是「母舌」，但它写的小说风格是「历史正剧」——对话时它会毒舌地挑剔你的情节漏洞，但它写出来的正文是严肃的历史叙事

**术语解释**

- **SKILL**（技能）：Agent 的可配置能力单元。每个 SKILL 定义了 Agent 在某个方面的行为方式。用户可以组合、自定义、分享 SKILL。
- **Project Level Style**（项目级风格）：整个项目统一的写作风格设定，Agent 生成的所有内容都会遵循。
- **Paragraph Level Style**（段落级风格）：在同一篇文章内对特定段落或场景应用不同的写作风格。

## 项目管理

### 多项目切换（Project Switching）

多项目切换是 CN 的核心功能之一。用户可能同时在写一本小说、一篇论文和几篇博客，需要在不同项目之间快速切换。

- 顶部 Top Bar 提供**项目切换入口**，点击后弹出项目列表
- 每个项目是一个独立的工作空间，包含自己的文件树、Knowledge Graph、Characters、Worldbuilding 等数据
- 切换项目时，编辑器状态（打开的文件、光标位置等）会被保存和恢复
- Agent 的 Memory 和上下文也跟随项目切换

**术语解释**

- **Project**（项目）：一个独立的创作单元，包含所有相关文件和数据。类似 VS Code 的 Workspace 概念。
- **Project Switching**（项目切换）：在不同项目之间切换的操作，类似手机上切换 App。

### 项目类型与模式（Project Types & Modes）

不同类型的内容创作流程完全不同。CN 通过「项目类型」而非通用界面来处理这种差异。

**核心原则：提供专业模式，但不框死用户**

- CN 提供几种预设的项目模式，每种模式有不同的默认工具集和大纲结构
- 但用户的需求不可能被模板框住——模式只是起点，用户可以自由调整
- Agent 可以辅助用户在模式之间切换或混合使用

**预设模式示例**

- **小说模式（Novel Mode）**：默认启用 Characters、Worldbuilding、Knowledge Graph、创作大纲（卷/章/场景结构）、情节线可视化、时间线
- **文案模式（Copywriting Mode）**：候选版本生成（「抽卡」模式更突出）、字数限制、目标受众设定、风格快速切换
- **论文模式（Academic Mode）**：引用管理、脚注、参考文献格式、逻辑结构大纲（摘要 → 论点 → 论据 → 结论）
- **通用模式（General Mode）**：最小化的默认配置，用户自己按需开启功能
- 用户也可以创建**自定义模式**，保存自己常用的工具组合和配置

**重要思路：引导而非限制**

很多用户可能像 Leeky 一样——不懂某个领域的专业流程，但确实想做这件事。项目模式的价值不仅是提供工具，更是**引导用户完成他们不熟悉的流程**。Agent 可以像编辑一样提问、建议、补充，帮助用户把一个粗略的想法变成一个完整的作品。

**术语解释**

- **Project Type / Mode**（项目类型 / 模式）：创建项目时选择的工作模式，决定了默认开启哪些工具、使用哪种大纲结构、Agent 的默认行为等。类似 VS Code 的「语言模式」，不同语言有不同的语法高亮和工具。

### 模板系统（Templates）

*待设计* — 需要进一步讨论：

- 项目模板（Project Templates）：预设的项目结构，如「长篇小说模板」「学术论文模板」「博客系列模板」
- 文档模板（Document Templates）：预设的文档格式，如「角色设定表」「章节大纲」「世界观条目」
- 用户是否可以创建和分享自定义模板
- Agent 是否可以根据用户描述自动生成模板

### 导入与导出（Import / Export）

CN 的核心理念：**万物皆文件，开放而非封闭**。不做 Notion 那种对导入导出不友好的封闭产品。

**底层逻辑：File-based（以文件为基础）**

CN 的内容底层就是文件形态，用户的创作成果不应该被锁在某个专有格式里。

**导出格式支持**

- **Markdown (.md)** — 最通用的纯文本格式，程序员和写作者都熟悉
- **Word (.docx)** — 适合提交给编辑、出版社、学校等正式场景
- **Plain Text (.txt)** — 最纯粹的文本，无格式干扰
- **PDF (.pdf)** — 适合最终成品的分享和存档
- 未来可扩展：ePub（电子书）、LaTeX（学术排版）等

**导入格式支持**

- 支持从上述格式导入内容到 CN 项目中
- Agent 可以辅助导入后的格式调整和内容优化

**Agent 辅助排版与优化**

Agent 不仅参与内容创作，还可以在导出环节辅助：

- 根据目标格式自动排版（如 Word 的标题层级、PDF 的页眉页脚）
- 润色、优化文本以适配不同输出场景
- 批量处理多文件导出

**术语解释**

- **File-based**（基于文件的）：产品的数据存储和组织方式以文件为基本单位，而非像 Notion 那样以数据库 Block 为基本单位。好处是数据天然可移植、可版本控制。

## 协作与版本

### 版本回溯（Version History） — V1 必做

版本回溯是第一个版本就必须具备的核心功能：

- 用户每次保存或 Agent 每次修改都会产生一个版本快照（Snapshot）
- 版本列表展示：Version 1 → Version 2 → ... → Version N
- 用户可以逐个回看每个版本的完整内容
- 支持 Diff 对比任意两个版本之间的差异
- 用户可以选择回滚到任意历史版本
- 版本可以被用户手动命名（如「初稿完成」「编辑修改后」「终稿」）

**术语解释**

- **Version History**（版本历史）：记录文档每一次变更的时间线，用户可以随时回到过去的某个状态。类似 Google Docs 的版本历史或 Git 的 commit 历史。
- **Snapshot**（快照）：某一时刻文档的完整副本，就像给文档拍了一张照片。
- **Rollback**（回滚）：把文档恢复到之前某个版本的操作。

### Fork（分支） — V1 可做

Fork 不仅仅是作者自己的功能——它是 CN 的社区协作基础设施，本质上类似 GitHub 的开源模型，但服务于创意写作场景。

**场景一：作者自己的 Fork（Author Fork）**

- 「我想试试另一种情节走向，但不想丢掉当前版本」
- Fork 出来的分支独立发展，不影响原版
- 作者可以在多个 Fork 之间对比、合并
- 对创意写作特别有价值——探索不同的故事走向而不必担心丢失

**场景二：读者/社区的 Fork（Community Fork）— 这才是核心**

这是 CN 区别于所有传统写作工具的关键特性。想象这个场景：

- 一部存在了很多年的小说，有大量读者。很多读者对结局不满意，或者觉得中间某个情节转折可以写得更好
- 在 CN 上，读者可以直接 Fork 原作品，写出自己认为更好的版本
- 就像 GitHub 上的开源项目——任何人都可以 Fork、修改、发布自己的版本

**具体的社区 Fork 机制**

- **Fork**（分叉）：读者点击「Fork」，获得原作品的完整副本，可以自由修改任何部分
- **Patch（补丁）**：读者只修改某一章或某一段，提交一个局部修改建议。原作者可以选择是否采纳（类似 GitHub 的 Pull Request）
- **Rewrite（重写）**：读者基于原作品的世界观和角色设定，重写整个故事或大幅重构情节线
- **Fork Tree（分叉树）**：所有 Fork 形成一棵树状结构，读者可以浏览不同版本的演变路径——「这个版本是从第 15 章开始分叉的，作者改了结局」
- **Attribution（署名溯源）**：每个 Fork 自动标注它 Fork 自哪个版本、原作者是谁，尊重原创

**这为什么重要？**

- 传统写作平台是「作者写、读者看」的单向关系。CN 让它变成**双向的创作生态**
- 一部好作品的世界观和角色可以被社区持续发展，就像开源软件一样——原作者写了 V1，社区贡献者可以写出 V1.1、V2、甚至完全不同的分支
- 这也是 CN 的**内容护城河**之一：用户不只是来写作的，还来参与社区共创，这种网络效应是其他工具很难复制的

**权限模型（Permission Model）**

- **公开作品（Public）**：任何人都可以 Fork
- **受限作品（Restricted）**：只有获得授权的用户可以 Fork
- **私有作品（Private）**：只有作者自己可以 Fork（即场景一）
- 原作者可以设置：是否允许 Fork、是否允许 Patch 提交、是否允许商业使用等

**术语解释**

- **Fork**（分叉）：从原作品复制出一个独立副本，之后各自独立发展。来自 Git/GitHub 的概念。就像一条路走到岔路口，分成两条路各走各的。
- **Patch / Pull Request**（补丁 / 合并请求）：对原作品的局部修改建议。读者提交修改，原作者审阅后决定是否合并到原版。就像给软件提交一个 bug 修复。
- **Fork Tree**（分叉树）：所有 Fork 版本之间的关系图。可以看到每个版本是从哪里分出来的、改了什么。
- **Attribution**（署名/归属）：自动记录每个 Fork 的来源，确保原创者得到尊重。

## 视觉风格

### 主题系统（Theme）

CN 必须支持**暗色主题（Dark Theme）**和**亮色主题（Light Theme）**，且两者互为镜像：

**配色原则：极简黑白，拒绝花哨**

- 核心只有两个颜色：**黑**和**白**
- 暗色主题：黑色背景 + 白色文字 + 白色高亮
- 亮色主题：白色背景 + 黑色文字 + 黑色高亮
- 切换主题时，黑白对调，其他一切保持一致

**颜色使用规则**

- **可以用颜色的区域**：Knowledge Graph（节点/连线着色）、Memory（标签分类）、Search Panel（结果类型区分）——这些场景中颜色承载信息功能
- **禁止用颜色的区域**：Agent 面板、编辑器主体、工具栏、侧边栏——保持纯粹的黑白极客风
- **绝对禁用的颜色**：紫色、蓝色、金色——这三个颜色是最典型的「AI 味」配色，CN 要和这种审美彻底划清界限
- 整体追求 Notion 式的纯粹感：干净、克制、信息密度高但视觉噪音低

### 字体方案（Typography）— 已确定

> ⚠️ **注意**：以下为当初的语音记录原文。最终决策为 `Lora + Source Han Serif SC`（衬线），详见 `01-tokens.css`。

**UI 字体（菜单、按钮、标签、工具栏等非正文区域）**

- 英文：**Inter** — 当下最流行的 UI 字体，Notion/Linear/Vercel 都在用，清晰锐利
- 中文：**思源黑体（Source Han Sans / Noto Sans CJK）** — Google 和 Adobe 联合出品，开源，覆盖中日韩

**编辑器正文字体**

- 默认中文字体：**思源黑体**（偏现代，干净利落）
- 用户可自由切换字体，包括思源宋体（偏文学）等其他选项
- 英文正文随 UI 字体使用 Inter

**等宽字体（代码/技术内容）**

- **JetBrains Mono** — 专为代码设计，辨识度极高

**术语解释**

- **Sans-serif**（无衬线字体）：笔画末端没有装饰的字体，如黑体。看起来现代、干净。Inter、思源黑体都是。

- **Design Token**（设计令牌）：把颜色、字号、间距、圆角、阴影等设计参数抽象成 CSS 变量（CSS Custom Properties）。换主题时只需改变量值，整个界面自动更新。CN 的 Token 命名规范：`--cn-{category}-{name}`（如 `--cn-bg-surface`、`--cn-radius-md`、`--cn-shadow-float`）。

## 为什么不做 Block？

详细设计参见 AI Native 内容架构——结构层与呈现层设计 页面。

### 核心决策：不做 Notion 式的 Block

Notion 的每一个段落、每一个标题、每一张图片都是一个独立的 Block，每个 Block 都是数据库里的一条记录。这对 Agent 来说是一个巨大的负担——Agent 每次写入内容，都要把文本拆成 Block、创建 Block、逐个保存。而 CN 的做法是：

- **底层存储**：连续文本 + 语义标注（spans），没有 Block 概念。Agent 可以直接 append 文本，不需要考虑 Block 结构
- **结构层**：从连续文本自动派生出一棵 AST（段落、标题、列表等），就像 VS Code 里 Tree-sitter 对代码做的事情，但 CN 对散文做
- **呈现层**：这棵 AST 被渲染成对外展示的视觉效果——**看起来就像 Notion 那样美观**，但底层没有 Block 的负担

### ProseMirror 增量更新是什么意思？

用一个类比来解释：

**Notion 的做法（Block 模式）**

假设 Agent 生成了 20 个段落的内容。Notion 需要：创建 20 个 Block 对象 → 在界面上创建 20 个独立的 React 组件 → 每个组件各自渲染。就像在页面上一下子放了 20 个独立的小窗口，浏览器很容易卡顿甚至白屏。

**CN 的做法（ProseMirror 增量更新）**

同样 20 个段落的内容。CN 只需要：在已有的一整篇文档里「插入文本」→ ProseMirror 自动算出哪些 DOM 节点变了 → 只更新变了的部分。就像在一张已有的画布上追加笔触，而不是重新绘制整张画。

"增量更新（Incremental Update）"的意思就是：**只处理变化的部分，不管文档有多长，每次更新的成本都很低**。

**那删除呢？**

同样的道理。用户删掉一段话、Agent 删掉三个段落——ProseMirror 不需要「找到对应的 Block 然后逐个删除」。它只需要：

1. 收到一个 Transaction（事务）：「从第 320 个字符到第 580 个字符，删掉」
2. ProseMirror 自动更新内部文档模型
3. 自动算出哪些 DOM 节点受影响，只移除/调整那几个节点

整个过程和「增量更新」是同一套机制——**Transaction（事务）是统一的操作单元**，不管你是插入、删除、替换、还是移动文本，都走同一条路。这正是不做 Block 的好处：没有「删除某个 Block」的概念，只有「修改文档的某个范围」，增删改查全都是同一个操作模型。

**术语解释**

- **ProseMirror**：CN 使用的编辑器引擎。它把整篇文档当作一个整体来管理，而不是拆成独立的 Block。CN 直接使用 ProseMirror 而非 TipTap 封装层，以获得对文档模型、Transaction、流式写入的完全控制力。
- **Incremental Update**（增量更新）：只更新变化的部分，不重新绘制整个界面。就像修改 Word 文档时，Word 不会把整篇文章重新布局，只重新布局受影响的部分。
- **Transaction**（事务）：一次操作的完整描述。比如「把第 10-20 个字符替换成新文字」就是一个 Transaction。ProseMirror 的所有编辑操作（插入、删除、格式化）都通过 Transaction 来执行。
- **DOM**（Document Object Model）：浏览器的界面元素树。每个文字、每个按钮都是 DOM 上的一个节点。更新 DOM 很耗性能，所以要尽量少更新。

## 阅读体验策略

### "开放包容，但原生体验最优"

Leeky 的思路：Notion 的护城河本质上是“用 Notion 做的内容只有在 Notion 上看才最好”，但代价是导出导入体验很差。CN 的策略是：

- **开放**：导出导入充分支持，File-based 底层，用户的内容永远不被锁死
- **体验护城河**：用户在 CN 里得到的阅读和创作体验是最好的——不是通过封锁数据，而是通过**体验本身足够好，让用户自愿留下**
- 具体体现：
    - 多渲染模式（编辑模式、沉浸式阅读模式、大纲模式）——同一份内容，多种视觉呈现
    - Agent 辅助排版——根据输出媒介自动优化排版
    - Design Token 精细控制——字体、行高、间距等视觉参数都经过精心调校
    - 导出时保持高保真——PDF/EPUB 导出的排版质量接近专业排版软件

### 具体怎么做：实现最佳阅读体验的路线图

光说「可以做」没用，下面是具体要做哪些事、按什么顺序做：

**第一步：Design Token 精调（Round 1-2，基础）**

这是成本最低、效果最明显的一步。具体做什么：

- 定义一套完整的排版变量（Typography Tokens）：正文字号、行高（line-height）、段落间距（margin）、标题层级比例、引用缩进等
- 参考值：正文 16-18px，行高 1.6-1.8，段落间距 0.8-1.2em，标题用 modular scale（模块化比例尺，每级标题按固定比例递增）
- 中英文混排优化：中文正文适合更大的行高（1.8+），英文可以稍紧凑
- 把这些变量抽成 CSS Custom Properties（CSS 自定义属性），换主题 / 换字体 / 换阅读模式时只改变量值

**工程师要做的事**：在现有 Design Token 系统里补充排版相关的 Token，然后让编辑器的 CSS 全部引用这些 Token，不写死任何数值。

**第二步：阅读模式切换（Round 3-4，核心体验差异化）**

这是让 CN 真正区别于其他产品的关键。具体做什么：

- 新增一个 Reading Mode（阅读模式）的切换按钮（可以放在 Top Bar）
- 阅读模式下：隐藏所有编辑 UI（光标、工具栏、选区高亮）、左右面板自动收起、正文居中显示、应用更宽松的排版参数
- 技术实现：ProseMirror 原生支持 `editable: false` 状态，切换时把编辑器设为只读 + 加载阅读模式的 CSS Token 即可
- 进阶：可以做段落聚焦——当前段落正常显示，其他段落变灰变淡，帮助用户集中注意力（详见 Zen Mode 章节）

**工程师要做的事**：创建两套 Token 配置（编辑模式 / 阅读模式），写一个 `useReadingMode()` hook 切换状态，ProseMirror 的 `editable` 属性 + CSS class 联动切换。

**第三步：多渲染管线（Round 5+，高级体验）**

同一份内容可以用完全不同的方式呈现。具体做什么：

- **大纲模式（Outline Mode）**：只渲染标题层级，折叠正文，适合长文档的结构浏览。从 AST 里只抽取 heading 节点渲染即可
- **类书籍分页模式（Paginated Mode）**：像 Kindle 一样分页翻阅。需要精确的文本测量来决定每页能放多少内容（这是 Pretext 库未来可能用到的场景）
- **演示模式（Presentation Mode）**：按章节/标题分割成幻灯片，适合作者分享故事大纲或做 pitch

**工程师要做的事**：定义一个 `RenderPipeline` 接口（渲染管线接口），每种模式实现这个接口。输入是同一棵 AST，输出是不同的 DOM 结构 + CSS。

**第四步：导出高保真排版（Round 5+，锦上添花）**

- PDF 导出使用 Puppeteer（无头浏览器）或专业排版引擎，复用阅读模式的 CSS Token 输出高质量排版
- ePub 导出生成标准电子书格式，嵌入 CN 的排版 Token
- Agent 辅助：导出前 Agent 自动检查排版问题（如孤行、标题断页等）并修复

**术语解释**

- **Typography Tokens**（排版令牌）：字号、行高、间距等排版参数的变量化表达。改一个变量，全局排版跟着变。
- **Modular Scale**（模块化比例尺）：标题大小按固定比例递增的方法。比如比例 1.25 意味着 H3=16px、H2=20px、H1=25px，看起来很和谐。
- **RenderPipeline**（渲染管线）：把同一份数据用不同方式画出来的通道。就像同一段视频，可以输出 4K 也可以输出 720p。
- **段落聚焦**：Zen Mode 的子功能，只高亮当前段落、其他内容淡化为 30% 透明度。帮助注意力集中。不是独立模式，统一在 Zen Mode 下管理。

### 技术难度总评

**总体评估：可行，且 CN 的架构天然有优势。**

- **第一步（Design Token 精调）**：难度低，1-2 周可完成，立竿见影
- **第二步（阅读模式）**：难度中，2-3 周，这是体验护城河的核心
- **第三步（多渲染管线）**：难度中-高，每种模式 1-2 周，但不急
- **第四步（导出排版）**：难度中，依赖第二步的 Token 体系，Round 5+ 再做

**总结**：CN 的三层分离架构（语义层 → 结构层 → 呈现层）其实比 Block 模型更容易做出优秀的阅读体验。因为同一份数据可以用不同的渲染管线去呈现，这恰恰是 Block 模型做不到或很难做到的事情（因为 Block 同时是存储和渲染单元，切换渲染模式意味着底层结构也要变）。

## Pretext 库评估（技术雷达）

**Pretext**（`@chenglou/pretext`）是 chenglou（React Motion / Reason 语言的作者）刚发布的纯 JS/TS 文本测量与布局库，2026-03-26 初始发布，目前版本 0.0.2。

**它是什么**

它绕过浏览器的 DOM 测量（如 getBoundingClientRect），自己实现了文本测量逻辑，支持渲染到 DOM、Canvas、SVG。核心价值：避免浏览器 layout reflow（最昂贵的性能操作之一）。

**CN 可能用得上的场景**

- 导出排版：PDF/EPUB 导出时的精确离屏文本测量（分页、页眉页脚定位）
- 沉浸式阅读模式：类书籍分页排版的精确文本布局
- Knowledge Graph：可视化中节点文字的精确测量
- Canvas 渲染模式：如果未来做超大文档的虚拟化渲染

**社区实测数据（2026-03-30）**

有开发者对 Pretext 的核心技术路线（Canvas `measureText()` + 二分查找行边界替代逐字符 DOM 测量）做了实际基准测试，结果如下：

- 纯 DOM（无优化）：冷启动 580.5ms，processParent 530.0ms
- Pretext v1（旧 fallback）：冷启动 526.3ms，processParent 467.3ms，285 次 fallback
- Pretext v2（修复后）：冷启动 274.2ms，processParent 218.3ms，零 fallback
- 核心结论：**总耗时提速 53%**，3166 个文本节点全部走 Canvas 路径，节省 172,814 次 DOM `getBoundingClientRect` 调用，多行文本 DOM 查询减少 ~95%

这组数据验证了 Pretext 的核心思路确实成立——Canvas 测量替代 DOM 测量在大型文本场景下提速显著。

**当前状态：确认引入**

- 编辑器核心（ProseMirror 日常写作和 Agent 流式写入）不依赖 Pretext，ProseMirror 自己管理文档渲染和光标定位
- Pretext 在 CN 中的落地场景明确：导出排版（PDF/ePub 离屏分页）、类书籍分页阅读模式、Knowledge Graph 节点布局、超大文档虚拟化渲染
- 社区实测数据（53% 提速、17 万次 DOM 查询节省）验证了技术路线的可行性，收益确定
- chenglou 的作品质量一贯过硬（React Motion、Reason），对库的长期维护有信心
- 引入节奏：Round 5 开始集成，优先用于导出排版和分页阅读模式。前期可以先在非关键路径（如导出预览）上试跑，验证生产稳定性后再扩展到阅读模式和 KG 节点布局

**术语解释**

- **Layout Reflow**（布局回流）：浏览器重新计算所有元素的位置和大小的过程。当你读取一个元素的宽高时，浏览器可能被迫重新布局整个页面，非常昂贵。
- **技术雷达（Technology Radar）**：一种跟踪新技术的方法。把技术分为四个象限：评估（看看）→ 试用（小范围试）→ 采纳（正式用）→ 暂缓（不用了）。

## 编辑器引擎选型：为什么选 ProseMirror 而不是 TipTap

CN 直接使用 ProseMirror 而不是它的上层封装 TipTap。这是一个经过深思熟虑的决策。

### 背景：TipTap 和 ProseMirror 的关系

TipTap 不是独立的编辑器引擎，它是 ProseMirror 的「封装层」。TipTap 把 ProseMirror 的底层 API 包装得更好用，提供了一套插件商店式的扩展系统和更方便的 React 集成。类比：ProseMirror 是发动机，TipTap 是套了一层仪表盘和方向盘的成品车。

### 选 ProseMirror 的 5 个原因

**1. CN 的架构需要完全控制力**

CN 已经定了「不做 Block、连续文本 + 语义层 + 结构层 + 呈现层」的三层分离架构。ProseMirror 的文档模型（一棵连续的 AST 树 + Transaction 事务系统）和这个架构天然契合。TipTap 的扩展系统会在中间加一层不必要的抽象。

**2. Agent 流式写入是 CN 的命脉**

Agent 生成内容时需要直接操作 Transaction——插入文本、替换范围、批量更新。用 ProseMirror 可以精确控制每一个 Transaction 的行为。TipTap 的 API 会把这些操作再包一层，遇到边界情况（流式中断、部分回滚）时反而更难处理。

**3. 多渲染管线需要直接访问文档模型**

CN 未来要做阅读模式、大纲模式、分页模式——这些都需要从同一棵 AST 树用不同方式渲染。ProseMirror 的 `doc` 节点树可以直接被多个渲染器消费，而 TipTap 的封装默认假设「一个编辑器实例 = 一种渲染方式」。

**4. TipTap 的价值在于「快速搭建标准编辑器」，但 CN 不是标准编辑器**

TipTap 的优势（快速搭建、插件生态、简单 API）对 CMS 、博客编辑器很有价值。但 CN 是写作 IDE，需要的是「完全掌控」而不是「快速搭建」。

**5. 成本和许可证**

ProseMirror 是完全 MIT 开源的。TipTap 的协作功能（Hocuspocus）和部分高级扩展需要 Pro 许可证。CN 未来做协作时可以用 Yjs（也是开源的）直接对接 ProseMirror，不受 TipTap 的商业限制。

### 为什么不选 Lexical（Meta/Facebook 出品）

Lexical 是现代架构、天生为 React 设计，但它的底层模型是节点式的（每个段落、标题都是独立节点），本质上就是 CN 明确拒绝的 Block 模型。而且生态比 ProseMirror 年轻很多，长篇小说场景没有经过验证。

### 用户手写体验不会受影响

ProseMirror 不仅对 Agent 写入最优，对用户手写也是最优。因为用户打字时的体验（光标行为、输入处理、中文 IME 输入法、选区管理）100% 由 ProseMirror 的 `prosemirror-view` 层处理——TipTap 从来不参与这一层。两者的手写体验完全相同，而 ProseMirror 反而能提供更精细的控制来打磨「写作手感」：

- **Typewriter Mode**（打字机模式）：光标始终保持在屏幕垂直居中，用 `scrollIntoView` + 自定义 `scrollMargin` 实现
- **Smart Quotes**（智能引号）：用 `inputRules` 插件自动转换引号样式
- **Markdown 快捷输入**：输入 `#`  自动变标题、`-`  变列表，用 `inputRules` 插件
- **段落聚焦**：当前段落正常显示、其他淡化，用 ProseMirror 的 `Decoration` 系统（Zen Mode 的子功能）

### Electron 环境强化了这个决策

CN 是 Electron 桌面应用，不是 Web 端。Electron 的渲染进程跑的就是 Chromium，ProseMirror 在其中的运行方式和在浏览器里完全相同。但 Electron 带来了额外优势：

- **单一引擎**：只有 Chromium，不需要兼容 Safari/Firefox 的 `contenteditable` 怪癖，中文 IME 行为完全可预测
- **主进程卸载重活**：超大文档的 diff、版本快照、全文搜索索引可以通过 IPC 扔给 Electron 主进程或 Worker 线程
- **原生全屏**：Zen Mode 可以用 `BrowserWindow.setFullScreen(true)` 实现真正的全屏沉浸，不是 Web 的 Fullscreen API
- **本地文件直读**：通过 Node.js 直接读写本地文件，保存速度毫秒级

**术语解释**

- **TipTap**：ProseMirror 的一个上层封装库。提供更简单的 API 和插件生态，适合快速搭建标准富文本编辑器。CN 的需求超出了它的设计范围，所以直接用底层的 ProseMirror。
- **Lexical**：Meta/Facebook 出品的现代富文本编辑器引擎。架构现代但底层是节点模型（接近 Block），与 CN 的连续文本架构不兼容。
- **Schema**：ProseMirror 的文档结构定义。定义了文档可以包含哪些节点类型（段落、标题、列表等）和它们的嵌套规则。类似数据库的表结构定义。
- **Plugin**：ProseMirror 的功能扩展机制。每个插件可以监听事件、修改 Transaction、添加装饰。加粗、斜体、历史记录、快捷键都是通过插件实现的。

## UI 组件库策略（Component Library Strategy）

CN 不从零开发所有 UI 组件，而是采用「Headless 原语库 + 自定义样式 + 品控流程」的策略，用最少的开发成本做出最高质量的 UI/UX。

### 选型决策：Radix + shadcn/ui 为主，React Aria 补位

**主力方案：shadcn/ui（基于 Radix Primitives + Tailwind CSS）**

shadcn/ui 不是一个 npm 依赖包，而是一套**可以直接复制到项目里的组件源码**。底层用 Radix Primitives 处理交互逻辑和无障碍，Tailwind CSS 处理样式。执行 `npx shadcn add button` 就把 Button 组件的完整 `.tsx` 源码复制到项目的 `components/ui/` 目录里，之后这些代码完全归 CN 所有，可以逐行调整。

**为什么选这个方案**

- **底层是 Radix**：CN V1 已经在用 `@radix-ui` 包，迁移成本为零
- **代码归你所有**：不是 `node_modules` 里的黑箱，而是项目里的 `.tsx` 文件，可以逐行调成 CN 的黑白极简风
- **覆盖面广**：Dialog、Dropdown Menu、Context Menu、Command Palette（cmdk）、Tooltip、Popover、Select、Tabs、Toggle 等几十个组件全覆盖
- **Notion/Linear/Vercel 都是同一路线**：Radix Primitives + 自定义样式
- **Tailwind 配合 Design Token**：CN 的 CSS Custom Properties 可以直接注入到 Tailwind 配置里，一处改全局变

**补位方案：React Aria（Adobe 出品）**

React Aria 比 Radix 更底层——它不提供组件，只提供 `useButton()`、`useDialog()`、`useMenu()` 这样的 Hook。无障碍支持是所有库里最强的（Adobe 在 Accessibility 上投入最大）。

- 80% 的组件用 Radix/shadcn/ui：Dialog、Popover、Tooltip、Select、Tabs、Switch、ScrollArea、Separator
- 20% 的高要求组件用 React Aria Hook：如果右键菜单或 Command Palette 需要更精细的键盘导航和无障碍控制，用 `useMenu()`、`useComboBox()` 替代 Radix 的对应组件
- 两者不冲突，可以在同一个项目中混用

**其他备选方案（已排除）**

- **Ark UI**（Chakra 团队）：底层用 Zag.js 状态机，架构更优雅但生态不成熟，社区案例少
- **MUI / Ant Design**：自带强烈设计语言（Material Design / 蓝色企业风），与 CN 的极简黑白风冲突，排除
- **完全自建**：几十个组件全部自建至少多 2-3 个月工期，Accessibility 和键盘导航细节极多，不划算

### 组件分配清单

**直接用 shadcn/ui（Radix 底层）的组件**

- **Dialog / Sheet** → Agent Side Panel、设置弹窗、确认对话框
- **DropdownMenu** → 右键上下文菜单、文件切换列表
- **Command（cmdk）** → Command Palette / 全局搜索
- **Tooltip** → 菜单项的 Hover 提示、工具栏按钮说明
- **Popover** → 翻译小面板、颜色选择器、字体选择器
- **Toggle / Switch** → Zen Mode 段落聚焦开关、Typewriter Mode 开关、设置项开关
- **Tabs** → 设置页面分类、Agent Panel 模式切换
- **ScrollArea** → 侧边栏、文件列表、Agent 对话区域的自定义滚动条
- **Separator** → 菜单分隔线、面板分隔线
- **Slider** → 字体大小调节、行高调节
- **Select** → 项目模式选择、字体选择、语言选择
- **Input / Textarea** → 设置页面的输入框、手动快照命名

**需要自建的组件（shadcn/ui 不覆盖）**

- **ProseMirror 编辑器本体** — 已决定直接使用 ProseMirror，是 CN 的核心
- **Floating Toolbar** — 选中文本的浮动工具栏，需要和 ProseMirror Selection 联动
- **Capsule Input（胶囊形 Agent 输入框）** — Zen Mode 的自定义形态，动态尺寸
- **Tool Orb（工具球）+ 展开工具栏** — Zen Mode 左上角的自定义 UI
- **Navigation Outline（导航大纲）** — 紧贴编辑器的自定义 Outline UI
- **Knowledge Graph 画布** — 用 React Flow 或 D3 构建的可视化组件
- **Diff View（差异视图）** — Agent 修改前后对比的自定义渲染
- **File Switcher List（文件切换列表）** — 自定义的文件切换交互

### 如何让 Agent 产出高质量组件代码

Agent（如 Claude Code、Cursor）的输出质量取决于它能读到的上下文质量。CN 需要建立一套**Agent 可消费的规范体系**，让 Agent 每次生成或修改组件代码时都有明确的约束和参照。

**1. CN Design System 规范文档（Agent Context）**

这不是给人看的设计稿，而是一份结构化的、Agent 可以直接消费的技术规范：

- 每个组件的 **TypeScript Props 接口**：所有合法的 prop 类型、默认值、枚举值
- 所有合法的 **variant 组合**：Button 有 `primary | ghost | danger` 三种变体，不允许其他值
- **Design Token 变量名**：规范中写的是变量名（如 `--cn-bg-surface`、`--cn-radius-md`、`--cn-shadow-float`），不是色值或数字。Agent 生成代码时引用变量名，不允许写死任何数值
- **间距/圆角/字号规则**：全部用 Token 表达。`padding` 用 `--cn-space-2`（8px），`border-radius` 用 `--cn-radius-lg`（8px），`font-size` 用 `--cn-text-sm`（14px）
- 该文档放在项目根目录的 `.cn/design-system.md`（或 `.cursor/rules/` 目录下），Agent 每次操作时自动加载

**2. 黄金样本（Golden Examples）— Agent 生成 + 用户审阅循环**

「打磨」不是指用户自己写代码，而是一个 **Agent 生成 → 用户审阅 → 自然语言反馈 → Agent 迭代** 的循环。用户不需要会开发，只需要能判断「好不好看、好不好用」并用自然语言给出反馈。

**具体流程**

1. **Agent 生成组件代码**：用 Claude Code / Cursor 生成组件的完整代码（`.tsx` + Tailwind + Framer Motion 动画），生成时引用 Design System 规范文档和 Radix 原语
2. **Agent 通过 Figma MCP 出设计稿**：Agent 调用 Figma MCP，在 Figma Design 中自动生成对应组件的视觉设计——包括所有状态（默认态、悬停态、按下态、禁用态、暗色主题态）。用户在 Figma 中直观看到组件长什么样，不需要启动开发环境
3. **Storybook 实时预览**：Agent 同时生成 Storybook Story，用户在浏览器中看到组件的真实运行效果（可交互、可切换状态、可切换主题）
4. **用户审阅并反馈**：用户同时看 Figma 设计稿和 Storybook 实时效果，用自然语言给出修改意见。例如：「这个圆角太大了」「hover 颜色太亮」「按钮按下去没有弹性感」「暗色模式下对比度不够」「这个 shadow 太重了，要轻盈一点」
5. **Agent 迭代修改**：Agent 根据反馈同时修改代码和 Figma 设计稿，用户再审阅。循环直到满意
6. **锁定为黄金样本**：当组件达到满意状态后，标记为 Golden Example。后续 Agent 生成其他组件时，指令中引用这些黄金样本的代码风格、Token 使用方式、动画参数，Agent 模仿已有的高质量代码

**需要打磨的 5 个核心组件**

- **Button** — 定义 CN 的 variant 体系、hover/press/disabled 状态、Framer Motion 动画参数
- **Dialog** — 定义浮层组件的标准：背景遮罩、圆角、shadow、进出动画
- **DropdownMenu** — 定义菜单类组件的标准：icon-only 菜单项、hover tooltip、stagger 动画、键盘导航
- **Tooltip** — 定义提示类组件的标准：深色背景、小字号、延迟显示
- **Input** — 定义输入类组件的标准：focus ring、placeholder 样式、错误状态

**为什么这个流程能保证质量**

- 用户虽然不写代码，但**审美判断力**是 Agent 不具备的——用户能感知到「这个按钮按下去手感不对」，这种感性判断只有人能做
- Figma 设计稿让用户在代码运行之前就能看到视觉效果，减少「做出来才发现不对」的返工
- Storybook 让用户看到真实的交互效果（动画、状态切换），比静态设计稿更准确
- 5 个黄金样本建立了 CN 的「风格基因」，Agent 后续批量生成的几十个组件都会模仿这套基因，保持全局一致性

**3. Storybook 作为可视化契约**

Storybook 是组件的「可视化测试环境」——每个组件都有一组 Story（场景），展示它在不同状态下的外观：

- 每个组件必须有完整的 Story 覆盖：默认态、悬停态、按下态、禁用态、暗色主题态、极端内容态（超长文字、空内容、中英文混排）
- Agent 生成新组件后，必须同时生成对应的 Storybook Story
- 开发者在 Storybook 中逐个检查所有状态——这是品控的视觉关卡
- Storybook 同时也是 Agent 的参考源：Agent 可以读取现有 Story 的代码，理解每个组件应该支持哪些状态

**4. 严格的 TypeScript 类型约束**

- 每个组件的 Props 用 TypeScript 严格定义，不允许 `any` 类型
- variant 用 union type 枚举所有合法值：`type ButtonVariant = 'primary' | 'ghost' | 'danger'`
- Agent 写出 `variant="blue"` 会直接编译报错——**类型系统是自动品控**
- 所有组件导出一个标准的 `ComponentProps` 类型，方便 Agent 在组合组件时做类型检查

**5. ESLint 自定义规则（代码级防护）**

- 禁止在组件中使用硬编码颜色值（必须用 Token）：`no-hardcoded-colors`
- 禁止在组件中使用硬编码间距值（必须用 Token）：`no-hardcoded-spacing`
- 禁止使用 `any` 类型：`@typescript-eslint/no-explicit-any`
- Agent 生成的代码如果违反这些规则，lint 阶段直接报错，Agent 必须修复

### 如何将组件变成高质量的 UI/UX（品质三层）

组件库给你的是「正确的交互逻辑」（键盘导航、无障碍、焦点管理），但视觉品质和体验感受需要额外三层工作。这三层才是 CN 和「套了个模板」的产品之间的本质差距。

**第一层：Design Token 精调（决定「质感」）**

Radix/shadcn 组件出来是「能用但平平无奇」的。CN 的质感来自 Design Token 的精心调校：

- 圆角用 `8px` 而不是 `4px`（Notion 风格的柔和感）
- shadow 用 `0 4px 16px rgba(0,0,0,0.12)` 而不是浓重的投影（轻盈浮层感）
- hover 背景色用 `#F7F7F7`（亮色）/ `#3A3A3A`（暗色）而不是刺眼的蓝色
- 过渡时间统一用 `120ms`（Notion 的标准节奏——快到不拖沓，慢到能感知）
- 字号层级：`12px`（辅助信息）、`14px`（正文/菜单项）、`16px`（标题）、`20px`（大标题），不超过 4 级
- 这些微调数值决定了用户的潜意识感受——「这个产品有设计感」还是「这个产品像模板」

**第二层：Framer Motion 微交互（决定「灵魂」）**

Notion 的 UI 之所以让人觉得「有灵魂」，不是因为组件多特别，而是因为每一个状态转换都有精心设计的动画。CN 的每个组件都需要加上微交互层：

- **菜单弹出**：`scale: 0.95 → 1`，`opacity: 0 → 1`，`duration: 0.12s`，`ease: [0.2, 0, 0, 1]`
- **按钮按下**：`scale: 0.98`，用 `whileTap` 实现，松开后 spring 回弹
- **面板展开**：子元素 stagger 动画，每个子元素延迟 `20ms` 依次出现
- **Tooltip 出现**：延迟 `300ms` 后 fade in，`duration: 0.1s`
- **页面切换**：content area 的 `opacity` + `y` 位移过渡，`duration: 0.15s`
- 这一层是 Radix 不提供的——Radix 只管「打开/关闭」的状态切换，CN 用 Framer Motion 定义「怎么打开、怎么关闭」的视觉表达

**第三层：像素级品控流程（决定「专业感」）**

- **Visual Regression Testing（视觉回归测试）**：使用 Chromatic（Storybook 官方服务）或 Percy 对每个组件的每个状态截图。Agent 每次修改代码后自动对比截图差异，1px 的偏移都能发现并标记
- **真机测试清单**：Windows 上的 ClearType 字体渲染效果、不同 DPI 缩放（100%/125%/150%）下的显示效果、不同窗口尺寸下的响应式表现。每个 Round 结束前必须过一遍
- **「5 秒法则」**：每个新组件完成后，让一个非开发者看 5 秒钟。如果无法立刻理解这是什么、怎么用——设计还不够好，需要迭代
- **暗色/亮色双主题同步检查**：每个组件在两套主题下都必须达到同等视觉品质，不允许「亮色模式精心调过、暗色模式凑合能看」

### 工程落地节奏

**Round 1（基础设施）**

1. 初始化 shadcn/ui，执行 `npx shadcn init`，配置 Tailwind + CN 的 Design Token
2. 把 CN 的 CSS Custom Properties（`--cn-*`）注入到 `tailwind.config.ts` 的 `theme.extend` 中
3. 通过「Agent 生成 → Figma MCP 出设计稿 → Storybook 预览 → 用户审阅反馈 → Agent 迭代」循环，打磨 5 个黄金样本组件（Button、Dialog、DropdownMenu、Tooltip、Input），建立 CN 的组件风格基线
4. 搭建 Storybook，为黄金样本编写完整的 Story
5. 编写 CN Design System 规范文档（`.cn/design-system.md`），作为 Agent 的 context

**Round 2（批量生成）**

1. 用 Agent 批量生成剩余 shadcn/ui 组件（Popover、Select、Tabs、Switch、ScrollArea、Separator、Slider 等），指令中引用黄金样本和规范文档
2. 为每个生成的组件编写 Storybook Story 并人工审查
3. 配置 ESLint 自定义规则，防止硬编码值
4. 配置 Chromatic/Percy 视觉回归测试

**Round 3+（自建组件）**

1. 开发 CN 独有的自建组件（Floating Toolbar、Capsule Input、Tool Orb、Navigation Outline 等）
2. 自建组件遵循与 shadcn/ui 组件完全相同的 Token 体系、动画体系、类型约束
3. 每个自建组件完成后加入 Storybook 和视觉回归测试

**术语解释**

- **Headless UI / Headless Primitives**（无头 UI / 无样式原语）：只提供交互逻辑（打开/关闭、键盘导航、焦点管理、无障碍属性）而不提供任何视觉样式的组件库。你需要自己写 CSS 来控制外观。Radix UI 和 React Aria 都是这一类。
- **shadcn/ui**：基于 Radix Primitives + Tailwind CSS 的组件集合。不是 npm 包，而是通过 CLI 工具将组件源码复制到你的项目中，代码完全归你所有。用户包括 Vercel、Supabase 等。
- **Storybook**：组件的可视化开发和测试环境。每个组件可以在独立的页面中查看所有状态（默认、悬停、禁用、暗色主题等），不需要启动整个应用。
- **Visual Regression Testing**（视觉回归测试）：自动对 UI 截图并与上一次截图对比，发现任何视觉差异。防止代码修改导致意外的 UI 变化。
- **Golden Examples**（黄金样本）：通过「Agent 生成 → 用户审阅 → 反馈迭代」循环打磨到完美的参考组件。用户不需要会写代码，只需要在 Figma 和 Storybook 中审阅视觉效果并用自然语言给出反馈。Agent 生成新组件时以这些为模板，确保风格一致性。

## Zen Mode（沉浸写作模式）

Zen Mode 是 CN 的极致写作体验，对标 iA Writer 的 Focus Mode，但增加了 Agent 协作能力。进入 Zen Mode 后，用户看到的是一整个全屏的编辑器，没有任何其他附加的东西。

### 设计哲学：不是另一个产品，是同一个产品的「沉静皮肤」

Zen Mode 的本质非常简单——它就是 CN 原生编辑器本身，只是套了一层皮肤。这层皮肤做的事情是：隐去所有非写作 UI，让屏幕上只剩下文字。

- **不是「模式切换」，是「皮肤叠加」**：底层是同一个 ProseMirror 编辑器实例、同一份文档、同一套状态。Zen Mode 只改变 CSS Token 和 UI 可见性，不改变任何功能逻辑
- **核心感受：沉静**。深色主题下就是全黑的背景 + 白色的文字，一眼看过去，用户的注意力直接落在编辑文字这件事上。没有面板、没有按钮、没有任何视觉噪音来分散注意力
- **工具必须「藏起来」**：胶囊框和工具球平时几乎不可见，只在用户主动召唤时出现。它们存在的前提是不破坏沉静感
- **参考标杆**：iA Writer 的 Focus Mode（当前句子/段落高亮，其余淡化为 30% 透明度）是「沉浸式写作」的标杆实现

### 进入 Zen Mode 后的界面

```jsx
┌──────────────────────────────────────────────────┐
│  ●  ← 工具球（左上角）                            │
│                                                  │
│          大面积留白                              │
│                                                  │
│      ────────────────────────────      │
│      正文内容区域（居中，宽度约 680px）      │
│      精调的排版 Token：更大行高、更宽松的      │
│      段落间距，让每一行文字都有呼吸感      │
│      ────────────────────────────      │
│                                                  │
│                                                  │
│                                                  │
│           ┌──────────────────┐               │
│           │  输入指令...          │               │  ← Agent 胶囊框
│           └──────────────────┘               │
│              （初始较小，动态扩展）               │
└──────────────────────────────────────────────────┘
```

进入 Zen Mode 后，整个 Workbench Shell 完全隐藏（Icon Bar、Left Panel、Right Panel、Top Bar、Status Bar 全部消失），用户看到的只有三个元素：

1. **全屏编辑器**（占据整个屏幕）
2. **胶囊形 Agent 输入框**（底部）
3. **工具球**（左上角）

### 1. 全屏编辑器

- 使用 Electron 原生的 `BrowserWindow.setFullScreen(true)` 实现真正全屏，隐藏系统标题栏
- 正文居中显示，左右留大量留白（内容宽度约 680px，参考 iA Writer 的排版宽度）
- 应用 Zen 专属的排版 Token：更大的字号（比如 20px）、更宽松的行高（1.9+）、更大的段落间距
- 可选开启段落聚焦：当前段落正常显示，其他段落淡化为 30% 透明度（用 ProseMirror 的 Decoration 系统实现）
- 可选开启 Typewriter Mode：光标始终保持在屏幕垂直居中，文档随输入向上滚动

### 2. 胶囊形 Agent 输入框（Capsule Input）

Zen Mode 下的 Agent 入口不是侧边栏，而是一个极简的横向胶囊形输入框，固定在屏幕底部。

**视觉设计**

- **形状**：胶囊形（pill shape）——两端圆弧、中间平直
- **位置**：固定在屏幕底部居中，距窗口底边约 32px
- **颜色**：单薄的半透明边框，背景微微模糊（backdrop-filter: blur），不用时几乎不可见
- **状态**：默认极淡显示，只看到一个淡灰色的 placeholder（如「输入指令...」）。用户点击或用快捷键激活后，边框变清晰，可以输入文字

**动态尺寸（Dynamic Sizing）— 核心交互**

胶囊框不是固定尺寸，而是跟随输入内容实时伸缩，本质体积很小，只在需要时才变大：

*初始状态*

- 默认宽度较小（约 280px），高度单行（44px），看起来就是一个小巧精致的胶囊
- placeholder 文字居中，整个胶囊极度克制，几乎融入背景

*阶段一：横向扩展*

- 用户开始输入文字后，胶囊框随文字长度**向两侧水平延伸**，始终保持底部居中
- 扩展是实时的、平滑的（CSS transition width，约 100ms），文字有多长胶囊就有多宽
- 横向扩展有一个**最大宽度上限**（约为屏幕宽度的 30%，即 400-420px），确保胶囊始终保持小巧克制，视觉上居中且美观
- 在最大宽度以内，胶囊始终保持单行高度（44px），形态就是一条横向的胶囊

*阶段二：纵向扩展*

- 当文字长度超过最大宽度后，胶囊框**自动换行并向上延伸高度**（底边不动，顶边上移）
- 每新增一行，高度增加一个行高（约 24px），过渡动画平滑（CSS transition height）
- 纵向扩展上限为 4 行（约 140px 高度），超出后内容区域变为可滚动（overflow-y: auto），胶囊不再继续长高
- 胶囊的圆角始终保持（border-radius 不变），整体从「扁平胶囊」自然过渡为「圆角矩形卡片」
- 底部对齐始终不变——胶囊向上生长，不会向下推挤

*视觉感受举例*

- 短指令（「润色」「翻译成英文」）：胶囊只有一小条，约 280-350px 宽，极度克制
- 中等指令（「把第三段改得更紧凑，节奏加快」）：胶囊横向展开到约 400px（接近最大宽度）
- 长指令（多行详细要求）：胶囊先到最大宽度约 400px，再纵向生长为 2-4 行的小卡片
- 发送后（按 Enter）：胶囊平滑收回初始尺寸（280px × 44px），回归极淡状态

**工程实现要点**

- 宽度驱动：监听 input 的 `scrollWidth`，用 `Math.min(scrollWidth + padding, maxWidth)` 计算实际宽度
- 高度驱动：当 `scrollWidth > maxWidth` 时，切换为 `textarea` 模式（或 contenteditable），监听 `scrollHeight` 驱动高度
- 所有尺寸变化使用 CSS transition（width 100ms、height 100ms），不允许跳变
- 发送后 reset：清空内容 → 宽度回到初始值 → 高度回到 44px，过渡动画统一 150ms

**交互**

- 用户在胶囊框内直接输入自然语言指令，按 Enter 发送给 Agent
- Agent 的输出直接作用于编辑器（插入、替换、修改），不弹出额外面板
- 处理完成后，胶囊框自动回到极淡状态，不干扰写作
- 如果用户需要深度对话，可以通过快捷键退出 Zen Mode 回到主界面的 Agent Panel

### 3. 工具球（Tool Orb）

左上角有一个非常小的圆球（约 28px 直径），是 Zen Mode 下唯一的工具入口。

**点击后的行为**

用户点击工具球，从圆球位置向右侧展开一条横向工具栏（Toolbar Strip）：

- **字体切换**：点击弹出字体列表，用户可在 Zen Mode 中实时切换编辑器字体（思源黑体、思源宋体、其他用户自选字体）
- **段落聚焦开关**：开启/关闭段落聚焦（当前段落高亮，其他淡化）
- **Typewriter Mode 开关**：开启/关闭打字机模式（光标保持屏幕居中）
- **退出 Zen Mode**：回到主界面

**视觉设计**

- 工具球默认极淡（和胶囊框一样，几乎不可见），鼠标悬停时微微变亮
- 工具栏展开动画：从圆球位置向右侧「生长」出来，约 200ms，ease-out 缓动
- 工具栏本身也是胶囊形（圆角横条），与工具球视觉语言一致
- 点击工具栏外的任何位置，工具栏自动收回圆球

### 4. 字体切换（Font Switcher）

这是 Zen Mode 独有的功能，让用户在写作时可以实时切换字体，找到最舒服的视觉感觉。

- 默认字体：思源黑体（已在视觉风格章节确定）
- 切换选项：思源黑体（现代）、思源宋体（文学）、用户自选字体
- 切换时，编辑器字体实时变化，带一个微妙的 fade transition（约 150ms）
- 字体切换只影响 Zen Mode 的显示，不会改变文档本身的字体属性

### 进入和退出

- **进入**：快捷键（Cmd+Shift+Z / Ctrl+Shift+Z），或通过 Top Bar / Command Palette 触发
- **退出**：按 Esc、或通过工具球的退出按钮
- **过渡动画**：进入时，所有面板同时淡出 + 收缩，编辑器平滑扩展填满屏幕，约 300ms。退出时反向播放
- **状态保持**：进入和退出 Zen Mode 时，光标位置、滚动位置、编辑内容完全不变。用户可以无缝在主界面和 Zen Mode 之间切换

### 实现方案（给工程师）

- Zen Mode 是同一个 ProseMirror 编辑器实例 + 不同的 CSS Token 和 UI 状态，不是新建一个编辑器
- 创建一个 `useZenMode()` hook，管理：是否全屏、是否段落聚焦、是否 Typewriter Mode、当前字体
- 进入时：调用 `BrowserWindow.setFullScreen(true)`，隐藏 Workbench Shell，加载 Zen 专属 CSS class
- 段落聚焦用 ProseMirror 的 `DecorationSet`：根据光标所在段落的 `pos` 范围，给其他段落添加 `opacity: 0.3` 的 CSS class
- 胶囊框和工具球是独立的 React 组件，只在 Zen Mode 下渲染

**术语解释**

- **Zen Mode**（沉浸写作模式）：CN 的全屏无干扰写作模式。参考 iA Writer 的沉浸写作体验，但增加了 Agent 输入能力、段落聚焦和字体切换。不引入独立的「Focus Mode」命名，段落聚焦是 Zen Mode 的内置子功能。
- **Capsule Input**（胶囊形输入框）：屏幕底部居中的动态尺寸胶囊输入框。初始很小（280px），随输入内容先横向扩展（最大约 30% 屏幕宽度，即 400-420px），再纵向扩展（最多 4 行后开始滚动），发送后收回原始尺寸。用于在沉浸写作时快速给 Agent 发指令。
- **Tool Orb**（工具球）：左上角的一个微小圆球，点击后向右展开横条工具栏。类似主界面的 FAB，但更克制、更小。
- **Typewriter Mode**（打字机模式）：光标始终保持在屏幕垂直中心，文档随输入向上滚动。模拟实体打字机的体验——纸往上走，光标不动。

## 平台策略与快捷键体系

### 平台优先级：Windows First

CN 的第一个版本优先支持 Windows 平台。虽然 macOS 是付费用户密度最高、最具商业价值的平台，但基于当前开发条件（Leeky 只有 Windows 设备），V1 先做 Windows，后续再适配 macOS。

**Electron 的跨平台优势**

Electron 天生跨平台，绝大部分代码 Windows 和 macOS 共用。真正需要平台适配的部分：

- 快捷键映射：Windows 的 Ctrl 对应 macOS 的 Cmd
- 窗口控制按钮：Windows 在右上角（最小化/最大化/关闭），macOS 在左上角（红绿灯）
- 字体渲染：macOS 的文本渲染天生更平滑，Windows 需要额外调校 ClearType 和字体 hinting
- 文件路径：Windows 用反斜杠 `\`，macOS 用正斜杠 `/`——Electron 的 `path` 模块自动处理

**适配成本评估**：核心功能代码 95% 以上共用，平台差异主要在 UI 细节层。V1 做好 Windows 后，macOS 适配预计 1-2 周。

### 快捷键体系（Keyboard Shortcut System）

CN 的快捷键分为三个层级，互不冲突：

**1. 全局快捷键（Global Shortcuts）— 在任何界面都生效**

- Ctrl+Shift+P：Command Palette（命令面板 / 全局搜索）
- Ctrl+Shift+Z / Ctrl+Shift+F11：进入/退出 Zen Mode
- Ctrl+S：手动保存当前文件
- Ctrl+,：打开设置
- Ctrl+N：新建文件
- Ctrl+Shift+N：新建项目
- Ctrl+Tab：弹出 File Switcher（文件切换列表）
- Ctrl+W：关闭当前文件
- Ctrl+B：切换 Left Side Panel 显示/隐藏
- Ctrl+J：切换 Right Side Panel（Agent Panel）显示/隐藏

**2. 编辑器快捷键（Editor Shortcuts）— 在编辑器获得焦点时生效**

- Ctrl+Z：Undo（撤销）
- Ctrl+Shift+Z（或 Ctrl+Y）：Redo（重做）
- Ctrl+F：当前文档内查找定位（只查找，不替换——替换交给 Agent）
- Ctrl+Shift+O：Quick Outline 弹窗（搜索标题并跳转）
- Ctrl+D：选中当前词
- Ctrl+L：选中当前行
- Ctrl+Shift+K：删除当前行
- Alt+Up/Down：移动当前行上下位置
- Ctrl+Enter：在当前段落下方插入新段落
- Tab / Shift+Tab：缩进 / 取消缩进

**3. Agent 快捷键（Agent Shortcuts）— 与 Agent 交互相关**

- Ctrl+K：在光标位置唤起 Inline Agent 输入框（选中文本时对选中内容操作）
- Ctrl+L：唤起 Agent Side Panel（右侧边栏对话）
- Esc：打断 Agent 当前操作 / 关闭面板

**设计原则**

- 所有快捷键必须可自定义（在设置中重新绑定）
- 快捷键冲突检测：用户绑定新快捷键时，如果与现有快捷键冲突，系统提示并让用户选择覆盖或取消
- 快捷键速查表：Ctrl+Shift+?（问号键）弹出完整的快捷键列表
- macOS 适配规则：Ctrl 全部映射为 Cmd，Alt 映射为 Option，其余不变

**工程实现**

- 使用一个统一的 `ShortcutManager` 管理所有快捷键注册、冲突检测、自定义绑定
- 快捷键配置持久化到本地 JSON 文件（`keybindings.json`），用户可以直接编辑该文件（类似 VS Code）
- 三个层级的快捷键用 priority 区分：全局 > Agent > 编辑器，同一按键组合被多个层级注册时，高优先级生效

**术语解释**

- **Modifier Key**（修饰键）：Ctrl、Shift、Alt 这类需要和其他键组合使用的键。Windows 上 Ctrl 是主修饰键，macOS 上 Cmd 是主修饰键。
- **Keybinding**（键绑定）：一个快捷键和一个操作之间的对应关系。比如 Ctrl+S 绑定到「保存」操作。

## 搜索系统（Search System）

### 核心决策：全局搜索为主，Agent 做替换

CN 的搜索不是传统编辑器的 Ctrl+F「查找替换」，而是一个全局化的搜索系统，覆盖用户工作空间的所有内容。

### 1. 全局搜索（Global Search）— Command Palette

**入口**：Ctrl+Shift+P 或点击 Icon Bar 中的搜索图标

**搜索范围**

- 当前项目的所有文件内容（全文搜索）
- 文件名和路径
- Knowledge Base 中的实体（角色、地点、设定等）
- Memory 条目
- Agent 历史对话

**搜索结果呈现**

- 结果按类型分组（文件、角色、设定、记忆等），每组显示匹配条数
- 每条结果显示：匹配上下文（前后各显示一行）、文件路径、最后编辑时间
- 点击结果直接跳转到对应位置，关键词高亮
- 支持模糊搜索（typo tolerance）——用户输入「角色」也能匹配到「角色设定」

**实现方案**

- 使用本地全文搜索引擎（如 FlexSearch 或 MiniSearch，纯 JS 实现，无外部依赖）
- 在 Electron 主进程维护搜索索引，文件保存时增量更新索引
- 搜索请求通过 IPC 发送到主进程，结果返回渲染进程，避免阻塞 UI

### 2. 当前文档内查找（Document Find）

**入口**：Ctrl+F

**功能**：在当前正在编辑的文档内查找关键词并高亮所有匹配位置

- 顶部弹出查找栏（类似 VS Code 的 Ctrl+F 栏），显示匹配数量和当前位置（如「3/12」）
- Up/Down 箭头或 Enter/Shift+Enter 在匹配项之间跳转
- 支持大小写敏感、正则表达式（高级选项，默认关闭）
- **不提供「替换」功能**——替换操作交给 Agent

**为什么不做传统替换**

CN 是 Agent Native 产品。传统的「查找替换」是机械的字符串匹配替换，而 Agent 可以做语义级别的替换：

- 「把所有'小明'改成'小亮'，但对话中别人叫他'明哥'的也要改成'亮哥'」——传统替换做不到，Agent 可以
- 「把第三章的所有被动语态改成主动语态」——这根本不是字符串替换，是语义操作
- Agent 执行替换后生成 Diff 预览，用户逐处确认或一键全部接受

### 3. Agent 辅助替换（Agent-Powered Replace）

**入口**：选中文本后 Inline Agent，或在 Agent Panel 中输入替换指令

**流程**

1. 用户描述替换意图（自然语言）
2. Agent 分析全文，找到所有需要修改的位置
3. 生成 Diff 预览（红绿标记），列出所有修改点
4. 用户可以逐条接受/拒绝，或一键全部接受
5. 所有修改作为一个 Undo 步骤，Ctrl+Z 一次性撤销

**术语解释**

- **全文搜索（Full-text Search）**：对文档的完整内容建立索引，支持关键词查询。不同于文件名搜索，全文搜索可以找到文档内部任何位置的匹配。
- **模糊搜索（Fuzzy Search）**：允许拼写不完全匹配也能返回结果。比如输入「角设」也能匹配到「角色设定」。
- **FlexSearch / MiniSearch**：纯 JavaScript 实现的全文搜索库，体积小、速度快，适合 Electron 本地使用。

## 数据持久化（自动保存、离线与云端策略）

### 自动保存（Auto-save）

**核心原则：无感、稳定、可靠。用户永远不需要想「我保存了没有」。**

**双重保存机制**

- **防抖保存（Debounce Save）**：用户停止输入 1.5 秒后自动保存。这是最常触发的保存方式——用户写完一句话停下来思考时，内容已经自动保存了
- **定时保存（Interval Save）**：每 60 秒强制保存一次，即使用户一直在打字不停顿。这是兜底机制
- **关键事件保存**：切换文件、切换项目、退出应用、进入/退出 Zen Mode 时立即保存

**保存速度**

- Electron 直接调用 Node.js 的 `fs.writeFile` 写本地文件，保存速度在毫秒级（10 万字文档保存 < 50ms）
- 保存过程完全异步，不阻塞 UI

**Status Bar 反馈**

- 保存中：显示小圆点动画（极其微妙，不干扰）
- 已保存：显示「已保存」+ 时间戳（如「已保存 · 10:32」）
- 未保存：显示小圆点标记，提示有未保存的更改

**崩溃恢复（Crash Recovery）**

- 每次自动保存时，同时写一份 `.recovery` 临时文件
- 应用崩溃后重新打开时，检测到 `.recovery` 文件，提示用户「检测到上次未正常关闭，是否恢复」
- 恢复文件保留最近 3 次，防止恢复文件本身也损坏

### 离线策略（Offline Strategy）

**V1：纯本地，无需联网编辑**

- CN V1 是 Electron 本地应用，所有文件存储在本地磁盘
- Agent 功能需要网络（调用 LLM API），但编辑器本身完全离线可用
- 网络断开时：编辑器正常工作，Agent 功能灰显并提示「网络不可用，Agent 暂时离线」

**V2+：云端同步（Cloud Sync）**

- **Local-first 架构**：本地是主副本（source of truth），云端是同步副本
- 用户在本地编辑 → 变更推送到云端 → 其他设备拉取变更
- 离线时照常编辑，恢复网络后自动同步
- 同步冲突处理：基于 CRDT（Conflict-free Replicated Data Type）或 OT（Operational Transformation）算法
- 云端存储方案待定（自建服务器 / AWS S3 / 其他），但接口层提前抽象，后续切换成本低

**术语解释**

- **Debounce**（防抖）：用户连续输入时不触发保存，停下来一段时间后才触发。避免每打一个字就保存一次。
- **Local-first**（本地优先）：数据先存本地，再同步到云端。即使没有网络，应用也能正常使用。与「云端优先」（如 Google Docs）相反。
- **CRDT**（无冲突复制数据类型）：一种数据结构，允许多个副本独立编辑后自动合并，不产生冲突。是实现离线同步和多人协作的核心技术。

## Undo/Redo 与版本管理

### Undo/Redo 系统

**基础能力：ProseMirror 原生 History 插件**

- 使用 `prosemirror-history` 插件，开箱即用
- Ctrl+Z = Undo（撤销），Ctrl+Shift+Z（或 Ctrl+Y）= Redo（重做）
- 操作粒度：ProseMirror 自动将连续的输入合并为一个 Undo 步骤（比如连续打了一个词，Undo 一次撤销整个词而不是逐字母撤销）

**Agent 操作与 Undo 的关系 — 关键决策**

Agent 的一次操作 = 一个 Undo 步骤。无论 Agent 修改了多少内容（插入一段话、替换三个段落、删掉一章），用户按一次 Ctrl+Z 就能撤销 Agent 的整个操作，恢复到 Agent 操作前的状态。

**工程实现**

- Agent 操作开始前，调用 ProseMirror 的 `tr.setMeta('addToHistory', false)` 将中间步骤排除出 history
- Agent 操作完成后，将整个变更打包为一个 Transaction 并标记为一个 history 步骤
- 如果 Agent 操作被用户中途打断，已插入的内容作为一个完整的 Undo 步骤保留（用户可以 Undo 撤销已插入的部分）

**History 深度**

- 默认保留 200 步 Undo 历史（覆盖当前编辑会话）
- 关闭文件后 Undo 历史清空——跨会话的恢复依靠 Version History（版本历史）

### Version History（版本历史）— 深化设计

已在「协作与版本」章节提到 V1 必做，这里补充具体实现方案。

**双轨制：自动快照 + 手动命名**

*自动快照（Auto Snapshot）*

- Agent 每次完成一个操作后，自动创建一个快照
- 每次手动保存（Ctrl+S）时，自动创建一个快照
- 每隔 30 分钟，如果文档有变更，自动创建一个快照
- 自动快照的命名格式：时间戳 + 操作描述（如「2026-03-30 22:15 · Agent 润色了第三章」）

*手动命名快照（Named Snapshot）*

- 用户可以随时通过快捷键（Ctrl+Shift+S）或菜单创建命名快照
- 弹出输入框，用户输入名称（如「初稿完成」「编辑第二轮」「终稿」）
- 手动快照在版本列表中用不同的视觉标记区分（加粗/标签色），方便快速定位

**版本浏览与回滚**

- 版本列表在 Left Side Panel 中展示（Icon Bar 添加一个时钟图标入口）
- 点击任一版本 → 在 Main Editor 中以只读模式预览该版本的完整内容
- 任意两个版本之间可以做 Diff 对比（红绿标记）
- 回滚操作：用户点击「恢复到此版本」，当前内容被替换为该版本内容，同时自动创建一个新快照保存回滚前的状态（防止误操作）

**存储方案**

- 底层使用增量存储（Delta Storage）：每个快照只保存与前一个快照的差异（diff），而非完整文件副本
- 这意味着 100 个快照的存储空间可能只比原文件大 2-3 倍，而非 100 倍
- 快照数据存储在项目目录下的 `.cn/history/` 文件夹中
- 自动快照保留策略：最近 7 天保留所有快照，7-30 天每天保留一个，30 天以上每周保留一个。手动命名快照永久保留

**术语解释**

- **Delta Storage**（增量存储）：只保存变化的部分，而不是每次都保存完整副本。类似 Git 的 packfile 机制。
- **Snapshot**（快照）：某一时刻文档的完整状态记录。虽然底层是增量存储，但用户看到的是完整的文档状态。

## 右键上下文菜单（Context Menu）

右键菜单是上下文感知的——在不同区域右键，弹出不同的菜单项。

### 视觉设计规范（Notion-style Custom Menu）

**参考标杆：Notion**

CN 的右键菜单对标 Notion 的自定义菜单风格——干净、有设计感、精致。不是 Windows 原生的粗糙方框，而是一个经过精心设计的自定义 UI 组件。

**菜单容器样式**

- 背景：亮色主题下纯白（`#FFFFFF`），暗色主题下深灰（`#2F2F2F`），带轻微的 `box-shadow`（`0 4px 16px rgba(0,0,0,0.12)`）营造浮层感
- 圆角：`border-radius: 8px`（Notion 风格的大圆角，柔和不锐利）
- 内边距：菜单容器上下 `padding: 6px 0`，每个菜单项 `padding: 6px 12px`
- 最小宽度：200px，最大宽度：280px
- 分隔线：1px 细线，颜色为极淡灰色（`rgba(0,0,0,0.06)` / 暗色主题下 `rgba(255,255,255,0.08)`），左右留 12px margin

**菜单项样式**

- 每个菜单项为一行，左侧是 **18px 图标**，右侧不显示文字——菜单项只显示图标
- 图标使用 CN 的统一图标库（线条风格，1.5px 线宽，与 Notion 的 Outlined Icon 风格一致）
- 悬停时：整行背景变为浅灰色（亮色主题 `#F7F7F7`，暗色主题 `#3A3A3A`），圆角 `4px`
- **Hover Tooltip**：光标悬停在任一菜单项上 300ms 后，在菜单项右侧或上方浮现一个小的文字提示气泡（Tooltip），显示该操作的**名称 + 快捷键 + 一句话解释**。例如悬停在剪刀图标上，Tooltip 显示：「剪切 · Ctrl+X · 剪切选中内容到剪贴板」。Tooltip 样式：深色背景 + 白色文字，`border-radius: 4px`，`padding: 4px 8px`，字号 12px
- 选中态/按下态：背景色加深一级，图标微微放大（scale 1.02），用 Framer Motion 的 `whileTap` 实现

**动画（Framer Motion）**

CN 的右键菜单动画使用 Framer Motion 库实现，不用原生 CSS transition：

- **弹出动画**：`initial= opacity: 0, scale: 0.95, y: -4`  → `animate= opacity: 1, scale: 1, y: 0` ，`duration: 0.12s`，`ease: [0.2, 0, 0, 1]`（Notion 的标准缓动曲线）
- **收起动画**：反向播放，`duration: 0.08s`
- **菜单项 stagger**：每个菜单项依次出现，stagger 间隔 `0.015s`，让菜单有「流水般」的入场感（Notion 的菜单就有这个细节）
- **子菜单展开**：从父菜单项右侧滑入，`initial= opacity: 0, x: -8`  → `animate= opacity: 1, x: 0` ，`duration: 0.1s`
- **Tooltip 浮现**：`initial= opacity: 0`  → `animate= opacity: 1` ，`duration: 0.15s`，延迟 300ms

**频率自适应排序（Frequency-based Adaptive Sorting）**

CN 的右键菜单会根据用户的使用频率自动调整菜单项的排列顺序，但有一条核心约束：

- **工具类菜单项（固定区）**：剪切、复制、粘贴、加粗、斜体、下划线等基础操作，位置永远固定不动。用户的肌肉记忆依赖这些操作在固定位置，移动它们会造成困惑
- **Agent 操作区（自适应区）**：润色、翻译、扩写、缩写、Ask Agent 等 Agent 相关操作，根据用户最近 1 天的使用频率自动排序——用得最多的排最上面
- 排序算法：简单的加权计数（最近 7 天权重 3x，7-30 天权重 1x），每次右键时重新计算排列
- 用户也可以在设置中关闭自适应排序，恢复为固定排列
- 排序变化时不做动画提示——悄悄调整，不干扰用户。用户会自然发现常用操作越来越顺手

**翻译交互（Translation Panel）**

翻译菜单项的行为不是直接调用 Agent，而是点击后**弹出一个翻译小面板（Translation Popover）**：

- 面板从翻译图标右侧滑出，宽度约 180px，高度自适应
- 面板内容：
    - **翻译为中文** — 一键点击，Agent 直接将选中文本翻译为中文
    - **翻译为英文** — 一键点击，Agent 直接将选中文本翻译为英文
    - 分隔线
    - **自定义翻译...** — 打开 Inline Agent 输入框，用户输入具体的翻译指令（如「翻译成日文并保持敬语」「翻译成法语口语风格」）
- 面板样式与主菜单一致（同样的圆角、阴影、Design Token），视觉上是主菜单的「延伸」
- 这两个快捷选项覆盖 90% 的翻译场景，只有特殊需求才需要打开 Agent 输入框

**术语解释**

- **Framer Motion**：React 生态中最流行的动画库。提供声明式的动画 API（`animate`、`initial`、`exit`），CN 的所有 UI 动画统一使用 Framer Motion，不混用 CSS transition 和 JS 动画。
- **Tooltip**（提示气泡）：鼠标悬停在某个元素上时弹出的小文字框，用于解释该元素的功能。CN 的右键菜单用 Tooltip 替代菜单项文字，既保持了图标的简洁感，又不丢失可发现性。
- **Stagger Animation**（错开动画）：多个元素依次出现，每个比前一个延迟一小段时间。视觉效果比所有元素同时出现更优雅。

**工程实现**

- 使用自定义 HTML/CSS 菜单（Custom Context Menu），不使用 Electron 原生 `Menu` API
- 参考标杆：Notion 和 Cursor（VS Code）都使用自定义菜单而非原生菜单，原因一致——原生菜单外观无法控制，与产品 Design Token 完全脱节
- 菜单组件统一封装为 `<ContextMenu />` React 组件 + `<ContextMenuItem />` 子组件，样式走 CN 的 Design Token 系统
- 动画引擎：**Framer Motion**（不用 CSS transition）。使用 `<AnimatePresence>` 管理菜单的挂载/卸载动画，`<motion.div>` 包裹每个菜单项实现 stagger 入场
- 菜单项根据当前上下文动态构建——监听 `contextmenu` 事件（`e.preventDefault()` 阻止浏览器默认菜单），判断触发位置和选中状态，渲染对应菜单项
- 弹出定位：菜单默认出现在鼠标点击位置的右下方。边界检测：如果菜单超出窗口边缘，自动翻转方向（右下 → 左下 → 右上 → 左上），确保菜单始终完整可见
- 弹出动画（Framer Motion）：`initial= opacity: 0, scale: 0.95, y: -4`  → `animate= opacity: 1, scale: 1, y: 0` ，duration 120ms，缓动曲线 `[0.2, 0, 0, 1]`。菜单项 stagger 间隔 15ms
- 收起：点击菜单外任意位置、按 Esc、或选择某个菜单项后，菜单 fade-out + scale 收缩，duration 80ms
- 键盘导航：Up/Down 切换高亮项，Enter 执行，Esc 关闭，Right 展开子菜单/子面板，Left 收起子菜单
- Tooltip 实现：使用 Framer Motion 的 `<AnimatePresence>` + `<motion.div>`，hover 300ms 后 fade-in 浮现，离开后立即消失。Tooltip 内容从菜单项配置中读取（`{ icon, label, shortcut, description }`）
- 频率排序：维护一个本地 `menuFrequency.json` 文件，记录每个菜单项的使用次数和时间戳。每次右键时读取并计算加权频率，决定自适应区的排列顺序。数据格式：`{ [actionId]: { count7d: number, count30d: number, lastUsed: timestamp } }`
- 翻译面板：`<TranslationPopover />` 组件，使用 Framer Motion 从触发位置右侧滑入（`initial= opacity: 0, x: -8` ），内部渲染快捷选项列表 + 自定义翻译入口 缓动
- 定位逻辑：菜单出现在鼠标点击位置，自动检测边界（靠近窗口右边缘时向左展开，靠近底部时向上展开）
- 子菜单（如「插入标题 → H1/H2/H3」）用 hover 延迟展开（150ms），避免误触
- 点击菜单外任意区域自动关闭（`useClickOutside` hook）
- 键盘支持：菜单打开后可用 Up/Down 箭头导航、Enter 选择、Esc 关闭
- 好处：外观与 CN 的整体视觉完全统一、可放入自定义组件（颜色选择器、Agent 输入框等）、跨平台外观一致、可做动画

**术语解释**

- **Context Menu**（上下文菜单）：右键点击时弹出的菜单。「上下文」的意思是菜单内容会根据你点击的位置和当前状态而变化。

## 通知与反馈系统（Notification System）

### 位置与形态

通知固定在窗口**右下角**弹出，采用 Toast 形态（从下方滑入的小卡片）。

**为什么是右下角**

- 用户的视觉焦点在编辑区（屏幕中央偏左），右下角是视觉盲区，通知出现在这里不会打断写作心流
- 如果放在顶部，每次通知弹出都会抢占用户视线——对沉浸思考中的创作者来说这是不可接受的
- VS Code 和 Cursor 都使用右下角通知，用户已经形成习惯

### 通知分级

**1. 系统级通知（System Notifications）**

- 应用更新可用
- 版本同步完成（V2+ 云端功能）
- 存储空间不足警告
- 样式：较醒目的边框，带图标，停留时间较长（8 秒）

**2. APP 级通知（App Notifications）**

- Agent 任务完成（「润色完成，已生成 Diff 预览」）
- 自动保存失败（「保存失败，请检查磁盘空间」）
- 导出完成（「PDF 已导出到桌面」）
- 样式：轻量卡片，停留 3-5 秒后自动消失

**3. 静默通知（Silent Notifications）**

- 自动保存成功（只更新 Status Bar，不弹 Toast）
- Agent 正在思考（只在 FAB 球上显示动画，不弹 Toast）

### Zen Mode 下的通知策略

- 所有非关键通知在 Zen Mode 下**完全静默**，不弹出任何 Toast
- 只有严重错误（如保存失败、文件损坏）才会以最小化形态在右下角微弱显示
- 退出 Zen Mode 后，错过的通知在通知中心中可查看

### 通知中心（Notification Center）

- 在 Status Bar 右侧有一个小铃铛图标，显示未读通知数量
- 点击弹出通知历史列表，用户可以查看所有最近的通知

**术语解释**

- **Toast**（吐司通知）：从屏幕边缘滑入的小型通知卡片，停留几秒后自动消失。叫「Toast」是因为它像烤面包机里弹出的面包片一样。

Icon Bar 最底部放置一个**齿轮图标**。用户点击后，设置界面以 **Full Editor View** 形态打开（替换主编辑区内容），就像 VS Code 的设置界面一样。

**1. 通用（General）**

- 语言切换（中文 / English / 未来更多语言）
- 启动行为（打开上次项目 / 显示项目列表）
- 自动保存间隔设置
- 自动更新开关

**2. 编辑器（Editor）**

- 默认字体选择
- 字号和行高调节
- Tab 宽度（2 / 4 空格）
- 自动换行开关
- 拼写检查开关

**3. Agent**

- Agent 人格设置
- 默认 SKILLS 启用/禁用
- Agent 响应语言偏好
- LLM 模型选择（如果支持多个模型）
- 上下文权限管理（Agent 可以访问哪些项目数据）

**4. 快捷键（Keyboard Shortcuts）**

- 完整的快捷键列表，每项都可以点击重新绑定
- 搜索框（快速查找某个操作的快捷键）
- 恢复默认按钮

**5. 外观（Appearance）**

- 主题切换（暗色 / 亮色）

**6. 文件与存储（Files & Storage）**

- 默认项目存储路径
- 自动备份设置
- 版本历史保留策略

**工程实现**

- 设置数据存储在本地 JSON 文件（`settings.json`），位于用户数据目录
- 使用 Zustand store 管理运行时设置状态，设置变更实时生效（不需要重启应用）
- 提供「打开设置文件」按钮，高级用户可以直接编辑 JSON（类似 VS Code 的 settings.json）

## 拖拽交互（Drag & Drop）

CN 的拖拽交互覆盖以下场景：

### 1. 文件树拖拽

- 拖拽文件到不同文件夹 → 移动文件
- 拖拽文件夹 → 移动整个文件夹及其内容
- 拖拽排序 → 在同一文件夹内调整文件顺序
- 拖拽到编辑器 → 打开该文件

### 2. 编辑器内容拖拽

- 选中文本后拖拽 → 移动文本到新位置
- 拖拽外部文件到编辑器 → 导入文件内容

### 3. 面板大小拖拽

- Left Side Panel 右边缘 → 调整侧边栏宽度
- Right Side Panel 左边缘 → 调整 Agent Panel 宽度
- 双击面板边缘 → 恢复默认宽度

### 4. 创作大纲拖拽

- 在创作大纲的 Full Editor View 中，拖拽章节/场景卡片 → 重新排序故事结构
- 拖拽是大纲编辑的核心交互之一

**视觉反馈**

- 拖拽过程中显示半透明的拖拽预览（drag ghost）
- 放置目标高亮显示蓝色指示线（drop indicator）
- 拖拽到无效区域时，光标变为禁止符号

**工程实现**

- 文件树和大纲拖拽：使用 HTML5 Drag and Drop API 或 `dnd-kit` 库（React 生态最成熟的拖拽库）
- 编辑器内容拖拽：ProseMirror 原生支持文本选区拖拽，无需额外实现
- 面板大小拖拽：CSS `resize` 属性 + 自定义 resize handle 组件

## Agent 流式输出与并发编辑（Streaming Output）

### 流式输出的视觉表现

**核心感受：文字像流水一样不断涌出，而不是一次性全部呈现。**

Agent 生成内容时，文字逐 token（词/字）流入编辑器，呈现打字机般的流动效果。用户可以实时看到 Agent「正在写」的过程，而不是等后台处理完毕后突然全部出现。

**具体表现**

- Agent 输出的文字以约 30-50ms/token 的速度逐个出现在编辑器中（实际速度取决于 LLM 返回速度，前端不人为延迟，但保证视觉流畅）
- 文字出现时带有微妙的淡入效果（opacity 0→1，约 80ms），让「涌出」的感觉更自然
- Agent 正在写入的区域有一个极淡的背景色标记（如 `rgba(0,0,0,0.03)`），让用户知道「这块区域正在被 Agent 修改」
- 写入完成后，背景色标记平滑消退（300ms fade out）
- FAB 球在 Agent 工作时显示呼吸动画，表示 Agent 正在思考/生成

**工程实现**

- 使用 Server-Sent Events（SSE）或 WebSocket 接收 LLM 的流式响应
- 每收到一个 token，通过 ProseMirror 的 Transaction 插入到文档指定位置
- 使用 `tr.setMeta('addToHistory', false)` 将每个 token 的插入排除出 Undo history
- 流式写入完成后，将整个操作打包为一个 history 步骤（用户 Ctrl+Z 一次撤销全部）

### 并发编辑策略：区域锁定

**用户可以在 Agent 生成的同时继续编辑——但有区域限制。**

Agent 正在写入的区域会被「软锁定」，用户不能在该区域内编辑（光标移入时显示禁止图标），但可以自由编辑文档的其他区域。

**为什么不完全禁止编辑**

- CN 是创作工具，用户的写作心流不应该被 Agent 打断
- Agent 可能在润色第三段，但用户突然想改第一段的一个错别字——这两个操作完全不冲突
- 完全禁止编辑会让用户感觉自己被工具「绑架」了

**为什么不完全允许编辑**

- 如果用户在 Agent 正在写入的段落内同时编辑，会导致位置偏移——Agent 的 Transaction 和用户的 Transaction 互相干扰
- 这不是不可解决的技术问题（CRDT 可以解决），但 V1 的复杂度太高，区域锁定是最稳妥的方案

**视觉反馈**

- Agent 正在写入的区域左侧显示一条细竖线（类似 Git diff 的变更标记），颜色为极淡灰
- 用户光标移入锁定区域时，光标变为「等待」图标，并显示 tooltip「Agent 正在编辑此区域」
- 锁定区域在 Agent 完成后立即解除，用户可以马上编辑

### 中途打断（Interrupt）

**必须允许用户随时打断 Agent 的操作。**

- **Esc 键**：按 Esc 立即停止 Agent 当前操作
- **Stop 按钮**：Agent 工作时，FAB 球变为 Stop 按钮（红色方块图标），点击停止
- **打断后的处理**：Agent 已生成的内容保留在编辑器中（不自动删除），用户可以保留或 Ctrl+Z 撤销
- **打断反馈**：Toast 通知「Agent 已停止」，Status Bar 显示「已中断」

**术语解释**

- **Streaming / 流式输出**：LLM 不是等全部生成完再返回，而是一边生成一边返回。每生成一个词就立刻发给前端，前端立刻渲染。这就是为什么 ChatGPT 的文字是逐渐出现的。
- **Token**：LLM 处理文字的最小单位。一个中文字通常是 1-2 个 token，一个英文单词通常是 1-3 个 token。
- **SSE（Server-Sent Events）**：一种让服务器持续向客户端推送数据的协议。LLM 的流式输出通常通过 SSE 传输。

## 错误处理与降级策略（Error Handling & Graceful Degradation）

### 核心原则：CN 不能因为没有 Agent 就丧失工作能力

CN 的使命是让用户在拥有 Agent 的情况下极大提升生产力，但**不代表用户没有 Agent 的时候就完全丧失生产力**。CN 首先是一个优秀的编辑器，Agent 是锦上添花的强大能力。

### 1. Agent 服务不可用

**触发场景**：网络断开、LLM API 超时、后端服务故障、API 配额用尽

**降级策略**

- 编辑器完全正常工作——用户可以打字、删除、格式化、保存、导出，所有手动编辑功能 100% 可用
- Agent 相关的 UI 元素灰显：FAB 球变灰、右键菜单中 Agent 操作项灰显、Inline Agent 入口隐藏
- Status Bar 显示 Agent 状态图标：绿色圆点 = 正常，灰色圆点 = 离线，红色圆点 = 错误
- 右下角 Toast 通知（只显示一次，不反复弹出）：「Agent 暂时不可用，编辑器正常工作」
- 网络恢复后自动重连，Toast 通知「Agent 已恢复」，所有 UI 元素恢复正常

**自动重试机制**

- Agent 请求失败后，自动重试 2 次（间隔 2 秒、5 秒）
- 3 次都失败后，停止重试，通知用户并提供「手动重试」按钮
- 不做无限重试——避免在服务完全宕机时消耗用户的网络和电池

### 2. 文件损坏与读取失败

**触发场景**：文件被外部程序修改、磁盘损坏、文件编码异常

**降级策略**

- 文件打开失败时，显示友好的错误提示页面（替代编辑器内容区域），包含：错误原因、建议操作（如从 Version History 恢复）、「打开恢复文件」按钮
- 自动尝试从 `.recovery` 文件恢复
- 如果 `.recovery` 也损坏，尝试从 Version History 最近的快照恢复
- 最坏情况：提示用户手动定位原始文件并尝试修复

### 3. 保存失败

**触发场景**：磁盘空间不足、文件被其他程序锁定、权限问题

**降级策略**

- 保存失败后立即 Toast 通知（红色，停留 8 秒）：「保存失败：磁盘空间不足」（附具体原因）
- Status Bar 保存状态变为红色警告
- 内存中的内容保持不变，不丢失任何数据
- 自动每 10 秒重试保存，直到成功或用户手动处理
- 提供「另存为」选项——让用户保存到其他位置

### 4. 超大文档性能降级

详见下一章节「超大文档性能」。

### 5. 通用错误处理原则

- **永远不丢数据**：无论发生什么错误，编辑器内存中的内容始终保持完整
- **永远给用户出路**：每个错误提示都附带至少一个可操作的建议
- **不弹烦人的弹窗**：使用 Toast + Status Bar 组合，不用模态弹窗（modal dialog）打断用户
- **错误日志**：所有错误写入本地日志文件（`.cn/logs/`），方便排查问题

**术语解释**

- **Graceful Degradation**（优雅降级）：当某个功能不可用时，系统不崩溃，而是自动降到一个较低但仍可用的状态。就像电梯坏了，楼梯还能走。
- **Modal Dialog**（模态弹窗）：一种霸占整个屏幕焦点的弹窗，用户必须处理它才能继续操作。非常打断心流，CN 尽量避免使用。

## 超大文档性能（Large Document Performance）

### 问题定义

对于文字创作者来说，一本长篇小说可能有 30-50 万字，一本学术著作可能有 20 万字以上。CN 必须在这种体量下保持流畅的编辑体验。

### 性能目标

- 10 万字以内的文档：打开 < 500ms，输入延迟 < 16ms（60fps），滚动完全流畅
- 10-30 万字文档：打开 < 1.5 秒，输入延迟 < 16ms，滚动流畅
- 30 万字以上文档：打开 < 3 秒，输入延迟 < 32ms（可接受），可能需要性能提示

### 技术方案

**1. ProseMirror 虚拟化渲染（Virtual Rendering）**

- 只渲染用户当前可见区域 + 上下各 2 屏的缓冲区（Buffer Zone）
- 超出缓冲区的文档内容只保留数据模型，不生成 DOM 节点
- 用户滚动时，动态加载/卸载 DOM 节点
- 这是超大文档性能的核心手段——30 万字文档在 DOM 中可能只有 2000-3000 个节点（而非全部渲染的 10 万+个节点）

**2. Electron 主进程卸载重活**

- 全文搜索索引、Diff 计算、版本快照生成——这些 CPU 密集型任务通过 IPC 扔给 Electron 主进程或 Web Worker 线程
- 渲染进程只负责 UI 渲染和用户交互，保证输入响应始终在 16ms 以内

**3. 文档分片加载（Lazy Loading）**

- 超大文档不一次性加载全部内容到内存
- 先加载文档元数据（标题结构、字数等）和当前光标附近的内容
- 其余内容在用户滚动到附近时按需加载

**4. 性能监控与提示**

- 内置性能监控：实时检测输入延迟、滚动帧率、内存占用
- 当文档体量接近性能瓶颈时（如 30 万字以上），Status Bar 显示提示：「大型文档模式 · 部分功能可能变慢」
- 提供「拆分文档」建议——Agent 可以辅助用户将超长文档拆分为多个文件（如按章节拆分）

**术语解释**

- **Virtual Rendering / 虚拟化渲染**：只渲染用户能看到的部分，看不到的部分不生成 DOM。就像一个超长的网页，浏览器不会一次性画完全部内容，而是画你能看到的那一屏。
- **Buffer Zone / 缓冲区**：在可见区域上下额外渲染一些内容，这样用户快速滚动时不会看到空白。
- **Web Worker**：浏览器/Electron 中独立于主线程的后台线程。可以做耗时计算而不卡 UI。

## Accessibility（无障碍）

CN 的无障碍支持确保视觉障碍、运动障碍、认知障碍用户都能使用产品。虽然 Leeky 没有这方面的经验，但这是一个负责任的产品必须考虑的维度，也是进入国际市场的合规要求。

### V1 必做（低成本高收益）

**1. 键盘完全可操作**

- 所有功能都必须能通过键盘触达，不能有只有鼠标才能操作的功能
- Tab 键在 UI 元素之间导航，Enter 键激活，Esc 键关闭/退出
- 焦点状态清晰可见——当前获得焦点的元素有明确的视觉指示器（focus ring）
- CN 已经规划了完整的快捷键体系，这本身就是键盘可操作性的核心

**2. 屏幕阅读器支持（Screen Reader）**

- 所有 UI 元素添加语义化的 ARIA 标签（`aria-label`、`aria-role`、`aria-describedby`）
- 编辑器内容使用正确的语义 HTML（`<h1>`-`<h6>`、`<p>`、`<ul>`、`<ol>` 等），ProseMirror 天然输出语义化 DOM
- Agent 操作完成时，通过 `aria-live` 区域朗读结果（如「Agent 已润色选中文本」）
- Icon Bar 和 Side Panel 的按钮都有 `aria-label`（如搜索图标标注「全局搜索」）

**3. 字号可调**

- 编辑器正文字号在设置中可调（12px-28px 范围）
- UI 字号跟随系统缩放设置（Electron 支持 `webFrame.setZoomFactor()`）
- Ctrl+= 放大、Ctrl+- 缩小、Ctrl+0 恢复默认

### V2 可做（进阶无障碍）

**4. 高对比度模式**

- 在现有黑白主题基础上，增加一个高对比度变体：更强的前景/背景对比、更粗的焦点指示器、更大的点击区域
- 高对比度模式是在 Design Token 层实现的——只需要一套新的 Token 值，不需要改组件

**5. 减少动画选项**

- 检测操作系统的「减少动效」偏好（`prefers-reduced-motion` 媒体查询）
- 开启后，所有过渡动画（面板展开、通知滑入、胶囊框伸缩等）变为即时切换，不做动画

**6. 语音输入集成**

- 未来可集成语音输入（Speech-to-Text），让运动障碍用户通过语音进行文字创作
- Electron 可以调用系统原生语音识别 API 或接入第三方服务

**术语解释**

- **ARIA**（Accessible Rich Internet Applications）：一套 HTML 属性标准，用于告诉屏幕阅读器页面上的元素是什么、做什么。比如一个图标按钮没有文字，但 `aria-label="搜索"` 告诉屏幕阅读器这是搜索按钮。
- **Screen Reader / 屏幕阅读器**：一种辅助软件，把屏幕上的文字和 UI 元素朗读出来，帮助视觉障碍用户使用电脑。Windows 上的 NVDA、macOS 上的 VoiceOver 都是屏幕阅读器。
- **Focus Ring / 焦点指示器**：键盘导航时，当前获得焦点的元素周围显示的一圈高亮边框。让用户知道「我现在在哪个按钮上」。

## 国际化（i18n — Internationalization）

### V1：中英双语

- CN V1 支持中文和英文两种界面语言
- 用户可以在设置中切换语言，切换后所有 UI 文字立即更新（不需要重启应用）
- 默认语言跟随系统语言设置（Windows 系统语言为中文则默认中文，英文则默认英文）

### 未来：多语言扩展

- 架构设计支持任意数量的语言扩展
- 优先级：中文 → 英文 → 日文 → 韩文 → 西班牙文 → 法文 → 德文（按目标用户规模排序）
- 考虑接入社区翻译贡献（类似 VS Code 的语言包机制）

### 工程实现

**所有 UI 字符串外部化**

- 不在代码中硬编码任何 UI 文字（如按钮文本「保存」「设置」等）
- 所有字符串提取到语言文件中（JSON 格式），按语言分文件：`zh-CN.json`、`en-US.json`
- 使用 `react-intl` 或 `i18next`（两个最成熟的 React 国际化库）管理翻译

**文字方向（RTL 支持）**

- V1 不需要支持 RTL（从右到左的语言，如阿拉伯语）
- 但在 CSS 层使用逻辑属性（`margin-inline-start` 而非 `margin-left`），为未来 RTL 支持预留空间

**日期和数字格式化**

- 使用 `Intl` API 处理日期、时间、数字的本地化格式
- 中文：2026年3月30日，英文：March 30, 2026

**Agent 语言**

- Agent 的对话语言独立于 UI 语言——用户可以设置 UI 为英文但 Agent 用中文回复，或反过来
- Agent 的响应语言在设置 → Agent → 响应语言偏好中配置

**术语解释**

- **i18n**：internationalization 的缩写（i + 18个字母 + n）。指的是让软件支持多种语言和地区的工程实践。
- **RTL（Right-to-Left）**：从右到左书写的语言，如阿拉伯语、希伯来语。支持 RTL 需要整个界面镜像翻转。
- **react-intl / i18next**：React 生态中最成熟的国际化库，提供翻译管理、日期格式化、复数处理等功能。

## 性能预算（Performance Budget）

CN 作为创作工具，性能直接影响写作手感。以下是各项核心指标的目标值。

### 核心性能指标

**1. 输入延迟（Input Latency）**

- 目标：< 16ms（60fps 帧时间）
- 含义：用户按下键盘到字符出现在屏幕上，不超过 16ms
- 这是写作工具最关键的指标——任何输入延迟都会破坏「手感」
- 参考：VS Code 的输入延迟约 10-15ms，Notion 约 30-50ms

**2. 应用启动时间**

- 冷启动（首次打开）：< 3 秒
- 热启动（已有缓存）：< 1.5 秒
- Electron 应用启动天生比 Web 慢，但可以通过懒加载和代码分割优化

**3. 文档打开时间**

- 1 万字以内：< 200ms（即时感）
- 10 万字以内：< 500ms
- 30 万字以内：< 1.5 秒
- 30 万字以上：< 3 秒 + 进度条

**4. 滚动帧率**

- 目标：稳定 60fps
- 在虚拟化渲染下，即使 30 万字文档也应保持 60fps 滚动

**5. Agent 流式输出帧率**

- Agent 流式写入时，编辑器渲染帧率不低于 30fps
- 每个 token 的 DOM 更新不超过 5ms

**6. 内存占用**

- 空载（无文档打开）：< 150MB
- 10 万字文档：< 300MB
- 30 万字文档：< 500MB

### 性能监控

- 开发阶段内置性能面板（开发者工具），实时显示输入延迟、帧率、内存占用
- 生产版本记录性能指标到本地日志，用于分析和优化
- 当检测到输入延迟持续超过 32ms 时，自动降级（如关闭段落聚焦动画等非必要渲染）

## 窗口管理（Window Management）

### 技术可行性：完全可行，难度中等

Electron 天生支持多窗口。每个窗口是一个独立的 `BrowserWindow` 实例，共享同一个 Electron 主进程。这意味着：

- 多个窗口可以访问同一份文件数据（通过主进程中转）
- 窗口之间的数据同步通过 IPC（进程间通信）实现
- VS Code 就是用这套机制实现多窗口的

### CN 的多窗口设计

**使用场景**

- 用户在编辑第 5 章的时候，想同时参考第 3 章的内容 → 两个窗口分别打开不同章节
- 用户在编辑正文的同时，想看 Knowledge Graph → 一个窗口编辑器，一个窗口 KG
- 用户在写作的同时，想看 Agent 的详细对话 → 一个窗口编辑器，一个窗口 Agent Full Panel

**窗口上限：最多 3 个窗口**

- 同一个项目内，用户最多同时打开 3 个窗口
- 3 个的理由：对文字创作者来说，同时参考 2-3 个文件已经是极端情况；再多窗口管理成本反而降低效率
- 超过上限时，新窗口替换最久未活跃的窗口（而非弹报错）

**同一文件多窗口编辑**

- 同一个文件可以在两个窗口中同时打开
- 一个窗口的编辑实时同步到另一个窗口（通过主进程转发 ProseMirror Transaction）
- 同步延迟 < 50ms，用户几乎感受不到

**工程实现**

- 每个 `BrowserWindow` 运行独立的渲染进程，各自有自己的 ProseMirror 编辑器实例
- 文档数据在主进程中维护一份「权威副本」（source of truth）
- 窗口 A 的编辑 → 通过 IPC 发送到主进程 → 主进程更新权威副本 → 通过 IPC 推送到窗口 B → 窗口 B 的 ProseMirror 应用 Transaction
- 这套机制和未来的多人协作（CRDT）架构天然兼容——多窗口同步本质上就是「单用户多设备协作」

**窗口管理 UI**

- 快捷键 Ctrl+Shift+N 打开新窗口
- 每个窗口的 Top Bar 右侧显示窗口编号（如「窗口 1/3」），方便用户区分
- 关闭最后一个窗口 = 关闭应用

**术语解释**

- **BrowserWindow**：Electron 中每个独立窗口的实例。每个窗口运行自己的 Chromium 渲染进程，但共享主进程。
- **IPC（Inter-Process Communication）**：进程间通信。Electron 的主进程和渲染进程之间通过 IPC 交换数据。类似两个人通过对讲机说话。

## 插件/扩展架构（Extension System）— TODO

### 当前决策：V1 不开放插件系统

- CN V1 专注于核心编辑体验和 Agent 能力，不开放第三方插件
- 原因：插件系统需要稳定的 API、安全沙箱、审核机制——这些在产品早期投入过高
- 参考：VS Code 在 1.0 才正式开放插件市场，之前经历了很长的内部 API 稳定期

### 架构预留

虽然 V1 不开放插件，但底层架构必须为未来的扩展性预留空间：

- **ProseMirror Plugin 体系**：编辑器的所有功能（加粗、历史记录、快捷键等）本身就是 ProseMirror Plugin 实现的。未来第三方插件可以用同样的机制扩展编辑器
- **SKILL 系统**：Agent 的 SKILLS 天然就是一种「AI 插件」。未来开放 SKILL 市场让用户分享和下载 SKILLS 是自然的扩展路径
- **模块化架构**：Icon Bar 的功能模块、Left Panel 的面板、Full Editor View 的工具都应设计为可注册/可卸载的模块。未来插件可以注册新的面板和工具
- **API 稳定化**：V1 期间持续沉淀内部 API 的设计，等到 API 足够稳定后再开放给第三方

### 未来可能的插件方向

- 自定义写作工具（如特定类型的字数统计、大纲生成器）
- 新的导出格式支持
- 第三方 LLM 模型接入
- 社区主题/字体包
- 特定领域的 Agent SKILLS（如学术引用管理、剧本格式化）

## 关键设计原则

1. **一眼可读（Glanceability）** — 界面即说明书，不需要教程。
2. **克制装饰（Restraint）** — 每一个视觉元素都必须有功能理由。
3. **Agent 多通道（Multi-channel Agent）** — 智能无处不在的同时，提供从 Inline 到全屏的多层级显式入口，但用户看到的永远是「Agent」而非「AI」。
4. **复杂内敛（Complexity Inside）** — 底层工程可以极其复杂，但用户永远感受到的是简单。
5. **参考而非抄袭** — 学习 Cursor / Notion 的设计语言，但服务于 CN 自身的创作场景。
