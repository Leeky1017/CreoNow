# CreoNow 最终架构设计书

> **定位**：本文件是 CN 项目的唯一架构决策文件，综合两份独立分析的最优判断。  
> **读者**：项目决策者与全体开发人员。  
> **基准日期**：2026 年 4 月。

---

## 一、CN 是什么

CN 是一个**桌面端 AI 辅助创作工具**，目标用户是小说家、编剧、内容创作者。

产品目标用一句话讲：

**让创作者敢长期依赖的 AI 写作工作台。**

"敢长期依赖"意味着三个不可退让的承诺：

1. **不伤稿** — AI 的任何修改都可预览、可确认、可拒绝、可回退。
2. **不失控** — 费用可预判，操作有记录，历史不丢失。
3. **越来越懂你** — 系统记住角色、设定和写作偏好，写得越多，AI 越顺手。

V1 完成时，用户的使用流程应该是：

```
打开 CN → 创建或打开创作项目
        → 在编辑器中写作
        → 遇到卡壳时，选中文字，让 AI 续写 / 润色 / 改写
        → AI 理解当前章节 + 角色设定 + 用户写作风格
        → 返回建议，展示差异
        → 用户一键接受或拒绝
        → 自动存版本快照
        → 继续写作
        → 随时回退到任意历史版本
        → 完成后导出为 Word 或 PDF
```

每一步都要做得**优秀**。衡量成功的标准不是功能数量，而是：用户愿不愿意每天打开 CN 来写作。

## 二、现状诊断

### 已完成的

- 15 个功能模块的详细设计文档（约 7000 行），单模块内部设计质量高。
- IPC 契约系统设计精良，可直接实施。
- 每个模块的异常处理和性能指标有初步定义。

### 三个核心问题

#### 问题 1：有房间图纸，缺整栋楼结构图

15 个模块各自设计精细，但缺少一个**编排层**来管理跨模块执行链路。

当用户说"帮我润色这段文字"，实际需要：

**识别意图 → 组装上下文 → 选模型 → 调 AI → 流式回传 → 展示差异 → 用户确认 → 写回文档 → 存版本快照 → 更新项目信息**

现在没有任何一个模块负责管理这整条链路。这是最大的架构缺口。

#### 问题 2：V1 想做太多

| 功能 | 现在的 V1 规划 | V1 真正需要的 |
|------|------------|------------|
| 版本管理 | 分支管理 + 三方合并 + 冲突解决 | 每次保存自动快照 + 一键回退 |
| 记忆系统 | 三层记忆 + AI 自动提炼 + 衰减公式 | 记住用户偏好和角色设定 |
| 知识图谱 | 图形化可视化 + AI 自动识别 + 5 万节点 | 角色和地点的列表管理 |
| 搜索功能 | 全文 + 向量 + AI 语义 + 融合排序 | 普通全文搜索 |
| 设计系统 | 三层架构 + 自动同步验证 | 一套统一样式文件 |

按现有规划开发，额外增加 40–60% 的开发周期，但对第一批用户几乎无感知价值。

#### 问题 3：关键基础机制未到位

- 统一编排层 — 缺失
- 原稿保护机制 — 无系统级方案
- 上下文预算管理 — 未定义（中文 token 估算尤其模糊）
- 跨模块失败处理 — 未制定规则
- API 费用控制 — 仅草案

## 三、从 Claude Code 学什么

CC 不是要复制的产品，而是一份**已被市场验证过的工程答案**。CN 和 CC 用户不同，但以下四条原则直接适用：

### 1. 先做管道，再挂功能

CC 的强不在功能名词多，而在于它先跑通了一条主管线：用户请求 → 上下文组装 → AI 调用 → 工具执行 → 结果呈现。所有能力都是插在管线上的插件。

**CN 必须先有自己的主管线，再把续写、润色、改写作为插件挂上去。不要先做 15 个模块再想办法拼在一起。**

### 2. 安全和确认是第一天就设计的

CC 从第一行代码就有权限检查和文件快照。对 CN 来说更重要——代码文件可以重写，创作原稿被 AI 改坏，用户信任直接崩塌。

**原稿保护不是功能，是系统级保障，必须从第一天内建。**

### 3. 上下文决定 AI 质量

AI 写作质量不取决于功能列表，而取决于上下文准不准、够不够、稳不稳。CN 最核心的上下文只有四项：

- 当前章节在写什么
- 项目的角色和设定
- 最近发生的剧情
- 用户偏好的写法

把这四项组装好，比任何高级功能都更能提升写作质量。

### 4. 第一版必须克制

CC 之所以走到今天，关键之一是它把复杂度拆成阶段，而非一次性做完所有未来能力。

## 四、核心架构决策

以下 6 条决策必须立即拍板，全部后续设计和开发以此为准。

### 决策 1：先后端闭环，再前端并进

**这是最关键的工程决策。**

如果后端主链路不稳定，前端会被迫反复重写。例：后端"润色"流程的接口一改，前端 AI 面板的展示逻辑全部返工。

正确顺序：

```
先把后端主链路做到 可调用 → 可验证 → 可保存 → 可回退
再让前端基于稳定接口接入
```

### 决策 2：编排层是地基的一部分

编排层（Orchestration Layer）不是阶段 3 的附加物，而是**阶段 1 就必须存在的系统脊柱**。

它负责：

- 接收用户请求并识别意图
- 组装上下文（当前章节 + 设定 + 偏好）
- 选择模型并调用 AI
- 流式回传结果
- 协调确认 → 写回 → 存版本
- 处理执行失败和回退

所有 AI 写作流程的总入口只有一个——编排层。

### 决策 3：原稿保护是系统级保障

任何会修改文档的 AI 操作，都必须：

1. 执行前自动创建版本快照
2. 结果先预览差异，不直接写入
3. 用户明确确认后才应用
4. 应用后仍可一键回退

版本快照不是"高级功能"，是所有写入行为的**默认安全垫**。

### 决策 4：每个模块区分 V1 和 V2

判断标准只有一条：**如果前 100 个用户大概率用不到，它就是 V2。**

V1 的唯一目标：做出一个安全、可回退、能长期使用的 AI 写作闭环。

### 决策 5：编辑器使用 ProseMirror

编辑器核心基于 **ProseMirror** 构建。

理由：

- ProseMirror 提供完全的底层控制能力，Schema 定义、Transaction 管理、Plugin 系统全部可自定义
- CN 的核心场景（AI diff 预览、选区级操作、版本快照集成、自定义文档结构）需要深度定制编辑器行为
- TipTap 本质是 ProseMirror 的封装层，在需要突破其封装边界时反而增加复杂度
- 创作工具的编辑器是产品命脉，必须握在自己手里而非依赖上层封装的设计决策
- ProseMirror 的 Transaction 机制与版本快照系统天然对齐——每次 Transaction 即可触发快照

### 决策 6：V1 验收标准是流程闭环而非模块完成率

V1 的验收标准是**"用户能连续完成创作流程"**，不是"模块列表完成度"。

具体地：一个用户可以打开项目 → 写作 → 让 AI 帮忙 → 看到差异 → 确认或拒绝 → 回退版本 → 导出成品。全程流畅、安全、可控。

## 五、四阶段开发路线

总原则：**先骨架后血肉 → 先闭环后铺面 → 先能用后好用 → 先验证后优化**

### 阶段 1：主骨架（后端闭环）

**目标**：最小可闭环系统——哪怕只有"润色"一个能力，系统也能从请求走到结果应用，全程安全可控。

**必须完成**：

| 项目 | 内容 |
|------|------|
| 编排层 | 统一入口，串联用户请求 → 上下文组装 → 模型调用 → 结果回传 → 确认 → 写回 → 存版本 |
| AI 写作主链路 | 至少"润色"能力端到端跑通 |
| 原稿保护 | 版本快照默认机制 + 预览/确认/拒绝/回退全链 |
| 上下文预算 | token 估算（中文场景）、上下文容量策略、优先级排序 |
| IPC 通信 | 基于已有契约设计实施进程间通信 |
| 基础数据层 | 文档存储、项目元数据、设定信息的持久化方案 |

**不做**：
- 前端完整 UI（此阶段前端仅做验证用的最小界面）
- 命令面板、主题切换、快捷键自定义
- 多模型切换策略
- 知识图谱、向量搜索
- 复杂记忆系统

**完成标准**：通过一个最小验证界面，可以触发润色请求 → 看到 AI 返回结果 → 确认写回 → 查看版本历史 → 回退。全部通过后端接口驱动。

---

### 阶段 2：前端接入 + 编辑器

**目标**：把"能跑"变成"用户能用"。前端围绕阶段 1 的稳定后端接口接入。

**必须完成**：

| 项目 | 内容 |
|------|------|
| 编辑器 | 基于 ProseMirror 的富文本编辑器（Markdown 语法支持、自动保存） |
| AI 面板 | 接入编排层接口，展示 AI 执行状态 |
| Diff 预览 | AI 结果与原文的差异展示（Inline Diff） |
| 核心技能 UI | 续写、润色、改写三个入口 |
| 基础操作 | 保存、恢复、取消、重试 |
| 费用感知 | 用户可见 AI 当前操作花了多少 token / 费用 |
| 基础布局 | 主窗口三栏布局 + 侧边栏 + 状态栏 |
| 统一样式 | 一套颜色、字体、间距规范（一个文件即可） |

**不做**：
- 分支版本管理、三方合并
- 大纲拖拽重排
- 多候选方案、AI 自动评价
- 自定义技能开发
- 主题切换

**完成标准**：用户打开应用 → 创建文档 → 写作 → 选中文字 → 选择润色/续写/改写 → 看到差异预览 → 接受或拒绝 → 浏览历史版本 → 回退。全程流畅。

---

### 阶段 3：项目与创作增强

**目标**：从"单文档写作"升级为"项目制长篇创作"。

**必须完成**：

| 项目 | 内容 |
|------|------|
| 项目管理 | 创建项目、项目内多文档管理、项目设定（类型/风格） |
| 设定管理 | 角色和地点的列表式 CRUD（不做图谱可视化） |
| 全文搜索 | 项目范围内的全文搜索（高频基础能力） |
| 简单记忆 | 记住用户写作偏好 + 项目级角色设定，AI 调用时自动带入 |
| 更多技能 | 一致性检查、对话生成、大纲扩写 |
| 导出 | Markdown / Word / PDF |

**不做**：
- 知识图谱可视化
- 向量语义搜索 / RAG
- 三层记忆架构 / AI 自动提炼
- 分支版本管理
- 自定义 AI 技能生态

**完成标准**：用户可以创建小说项目 → 添加角色和地点 → AI 续写时自动考虑已有设定 → 全文搜索 → 导出成品。

---

### 阶段 4：智能增强（AI 真正「懂你」）

**前提**：Phase 1-3 闭环已稳定运行。Phase 4 的目标是让 AI 从「通用写作助手」进化为「懂你项目的专属创作伙伴」。

> 「善战者，求之于势，不责于人。」——《孙子兵法》

**必须完成**：

| 项目 | 内容 |
|------|------|
| 向量检索 + RAG | 段落级嵌入 + 语义搜索 + 检索增强生成；AI 写作时自动检索相关章节和设定 |
| 三层记忆架构 | Working Memory（会话级）→ Episodic Memory（事件级）→ Semantic Memory（知识级）；AI 自动提炼和衰减 |
| 知识图谱可视化 | 力导向布局的角色关系图；时间线视图；自动实体提取和关系推断 |
| 高级上下文压缩 | 三层压缩策略（synopsis → micro-compact → auto-compact）；87% 窗口阈值触发；3 次失败熔断 |
| 命令面板 | Cmd/Ctrl+P 命令面板；快捷键自定义；最近文件快速跳转 |
| 主题系统 | 亮色 / 暗色 / 跟随系统三档切换；主题持久化 |

**不做**：
- ❌ 多候选生成 + AI 评审（Phase 6）
- ❌ 自定义 Skill 生态 / Skill 市场（Phase 6）
- ❌ 协同编辑（Phase 8）
- ❌ 跨项目联邦搜索（Phase 7）

**完成标准**：AI 续写小说时，自动检索前文相关章节 → 引用角色图谱中的关系 → 根据记忆调整风格 → 长对话自动压缩不卡顿。用户感受到"AI 越来越懂我的故事"。

---

### 阶段 5：产品打磨（从「能用」到「爱用」）

**前提**：Phase 4 的智能增强已稳定。Phase 5 聚焦打磨用户体验细节，让产品从"功能完整"变为"用户愿意每天打开"。

> 「文章千古事，得失寸心知。」——杜甫

**必须完成**：

| 项目 | 内容 |
|------|------|
| Onboarding 引导流 | 首次打开引导（创建项目 → 写一段 → 触发 AI → 体验闭环）；空状态优化 |
| 性能优化 | 代码分割 + 懒加载；虚拟滚动（长文档/长列表）；ProseMirror 大文档渲染优化（10 万字级） |
| 无障碍（a11y） | WCAG 2.1 AA 标准；键盘导航完整覆盖；屏幕阅读器支持 |
| 大纲视图 + 拖拽 | 文档大纲侧栏；章节拖拽重排；大纲与编辑器双向联动 |
| 分支式版本管理 | 版本分支 + 三方合并 + 冲突解决 UI；从线性历史升级为 DAG 图结构 |
| 多端导出增强 | ePub 格式支持；排版模板（小说、剧本、论文）；批量导出 |
| 错误恢复增强 | 组件级 Error Boundary；网络故障离线队列；自动重试策略 |

**不做**：
- ❌ 自定义 Skill 开发框架（Phase 6）
- ❌ 多语言界面 EN（Phase 7）
- ❌ 插件系统（Phase 7）

**完成标准**：新用户 5 分钟内完成首次 AI 写作体验；10 万字文档打开不卡顿（< 2s）；无障碍审计通过 WCAG AA。

---

### 阶段 6：AI 深度能力（从「助手」到「审稿人」）

**前提**：Phase 5 产品打磨完成。Phase 6 释放 AI 的深度能力——不只是续写，还能审、能评、能教。

> 「读书破万卷，下笔如有神。」——杜甫

**必须完成**：

| 项目 | 内容 |
|------|------|
| 多候选生成 + AI 评审 | 同一输入生成 3-5 个候选 → AI Judge 评分 → 用户选择最优 |
| Judge 系统 | 一致性审稿（角色行为一致性、时间线矛盾、伏笔回收）；风格评分 |
| 自定义 Skill 开发框架 | SKILL.md frontmatter 规范；Skill 参数化模板；Skill 组合链 |
| Agentic Loop 增强 | AI 写作中自主调用工具（查 KG、查记忆、查前文）；tool-use 循环优化 |
| 写作风格学习 | 基于用户历史文本训练风格偏好模型；AI 自适应模仿用户笔法 |
| 对话脚本编辑器 | 专为剧本 / 对白场景优化的编辑模式；角色标注 + 台词生成 |

**不做**：
- ❌ 多人协作（Phase 8）
- ❌ Skill 市场发布（Phase 7）
- ❌ 云端同步（Phase 8）

**完成标准**：用户选中一章 → AI 生成 3 个续写方案 + 评分理由 → 一致性审查报告指出"第 5 章张三的眼睛颜色与第 2 章矛盾"。

---

### 阶段 7：生态与国际化（从「产品」到「平台」）

**前提**：Phase 6 核心 AI 能力已稳定。Phase 7 将 CN 从"一个工具"扩展为"一个平台"。

> 「海纳百川，有容乃大。」——林则徐

**必须完成**：

| 项目 | 内容 |
|------|------|
| Skill 市场 | Skill 发布 + 分发 + 评价系统；开发者文档 + SDK |
| 插件系统 | 第三方插件 API（编辑器扩展、面板扩展、导出格式扩展）；沙箱安全 |
| 多语言界面 | 英语 / 日语 / 韩语 UI 翻译；i18n 框架已就绪，补全翻译文件 |
| 跨项目联邦搜索 | 跨项目语义搜索 + 引用；世界观共享（多部作品共用设定） |
| 高级设定管理 | 角色关系编辑器（可视化）；设定继承和模板；设定版本控制 |
| 创作分析面板 | 写作数据统计（字数/时间/AI 使用率）；创作习惯分析；写作热力图 |

**不做**：
- ❌ 实时多人协作（Phase 8）
- ❌ 云端存储（Phase 8）
- ❌ 移动端（超出桌面产品范围）

**完成标准**：第三方开发者可以发布自定义 Skill 到市场；用户可以安装 Skill 并立即使用；英语用户可以流畅使用全部功能。

---

### 阶段 8：协作与云端（从「单人」到「团队」）

**前提**：Phase 7 生态建立。Phase 8 突破单机桌面的边界，进入协作时代。

> 「独行快，众行远。」——非洲谚语

**可考虑的方向**：

| 项目 | 内容 |
|------|------|
| 实时协作编辑 | 基于 CRDT/OT 的多人实时编辑；光标可见；冲突自动解决 |
| 云端同步 | 项目云端备份 + 多设备同步；端到端加密（创作数据隐私） |
| 团队工作区 | 组织 / 团队 / 项目权限；编辑建议 + 审阅流程 |
| AI 编辑助手 | AI 以"编辑"角色参与协作；可标注建议、可讨论、可追踪 |
| 发布管道 | 从 CN 直接发布到连载平台 / 出版社排版系统 |

**关键原则**：Phase 8 是**远景规划**，不承诺时间线。只有当 CN 作为单机产品已被市场验证，且有明确的团队协作需求信号时，才启动。每一项都是独立产品级的工程挑战，不可低估。

## 六、绝对不要做的 4 件事

### 1. 不要按"模块平均主义"开发

不是每个模块都应该同时启动。决定优先级的标准只有一个：**它是不是主链路的一部分。**

### 2. 不要先做复杂前端，再等后端成熟

这会导致前端不断跟着接口变化返工，时间浪费在"重做"而不是"推进"上。

### 3. 不要把高级能力伪装成 V1 基础能力

如果一个功能必须依赖大量推理、复杂数据结构、复杂回滚、复杂调度，它大概率不是 V1 的必须品。

### 4. 不要在没有验证前，把复杂设计当成正确设计

复杂的 Memory、复杂的 KG、复杂的 Search 不自动等于更好的产品。对 V1 来说，**被用户持续使用** 比 **设计上看起来先进** 重要一百倍。

## 九、技术栈确认

| 层级 | 选型 | 理由 |
|------|------|------|
| 桌面框架 | Electron（或 Tauri 评估中） | 跨平台桌面应用标准选择 |
| 编辑器核心 | **ProseMirror** | 完全底层控制，Schema/Transaction/Plugin 可定制，与版本快照天然对齐 |
| 前端框架 | React | 生态成熟，组件化开发 |
| 进程通信 | IPC 契约系统 | 已有优秀设计，直接实施 |
| AI 调用 | 模型网关层（封装多模型切换） | 隔离模型差异，统一调用接口 |
| 数据存储 | 本地优先（SQLite 或文件系统） | 创作数据不上云（除非用户主动选择） |
| 版本管理 | 基于快照的线性历史 | V1 不做分支，只做保存点 + 回退 |

## 十、本文件的使用规则

1. **所有后续设计文档**必须与本文件的 6 条核心决策保持一致。
2. **任何模块的 V1 设计**必须通过以下检验：它是否属于"用户连续完成创作流程"所需的一环？如果不是，标记为 V2。
3. **开发排期**严格按阶段 1 → 2 → 3 → 4 的顺序推进。每个阶段结束时必须有可运行的产品，而非半成品集合。
4. **新增功能提案**必须回答：它解决的是前 100 个用户的核心痛点，还是未来的扩展想象？
