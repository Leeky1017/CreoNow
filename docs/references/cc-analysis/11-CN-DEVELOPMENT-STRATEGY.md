# CreoNow 开发战略：从何处来，到何处去 Opus版

> 一份写给决策者的非技术文档
> 

---

## 前言：这份文档是什么

这份文档回答一个问题：**CreoNow（以下简称 CN）作为一个 AI 创作 IDE，到底应该怎么开发？**

我们花了大量时间研究了 Claude Code（以下简称 CC）的完整源码（51 万行代码），也逐行审读了 CN 的全部 15 个设计规格书（约 7000 行设计文档）。这份报告基于这些研究，用非技术人员也能理解的方式，给出一条清晰的开发路线。

---

## 第一部分：我们面对的现实

### 1. CN 是什么

CN 是一个**桌面端 AI 辅助创作工具**。它的目标用户是作家、编剧、内容创作者。用户安装后，可以在里面写小说、剧本或其他长篇内容，AI 会在旁边帮助润色、续写、检查一致性等。

类比：如果说微软 Word 是普通文字处理器，那 CN 就是"AI 加持的创作专用 Word"。

### 2. CC 是什么

CC 是 Anthropic 公司为程序员做的 AI 编程工具，已经是市场上最成功的 AI 编程助手之一。它的代码完全公开，我们可以学习它的设计思路和工程经验。

类比：CC 就像一个非常优秀的"参考答案"——不能直接抄（因为面向不同用户群体），但可以学习它的解题思路。

### 3. 现在 CN 的规划做到了什么程度

**已经完成的（规划层面）**：
- 15 个功能模块的详细设计文档（覆盖了编辑器、AI 服务、版本管理、知识图谱等几乎所有功能）
- 模块间通信的标准化方案（IPC 契约系统，这部分设计得非常出色）
- 每个模块的异常处理和性能要求

**还没有做到的（架构层面）**：
- 模块之间如何协同工作的总体方案还不完善
- 很多模块的 V1 版本定义了过多功能，可能导致开发周期过长
- 部分关键的技术验证还未进行

---

## 第二部分：存在的核心问题

### 问题 1：盖房子画了精美的房间图纸，但缺少整栋楼的结构图

**通俗解释**：CN 的 15 个模块就像 15 个房间，每个房间的装修方案都做得非常精细（墙面颜色、家具款式、灯光方案都有），但整栋楼的框架结构——承重墙在哪里、水电管道怎么走、各层楼板如何连接——设计不够。

**具体表现**：
- 当用户说"帮我润色这段文字"时，需要经过 4 个模块的接力配合（技能路由 → 上下文组装 → AI 调用 → 编辑器应用），但没有一个模块负责管理这整个流程
- 如果其中某一环出错了，没有明确的处理机制

**影响**：开发时容易出现"每个模块单独测试都没问题，但组合在一起就出错"的情况。

### 问题 2：第一版想做太多事情

**通俗解释**：就像开一家餐厅，第一天就想同时提供中餐、西餐、日料、烧烤——每样都需要不同的厨房设备和厨师团队，结果可能每样都做不好。

**具体表现**：

| 功能 | 现在的 V1 规划 | V1 真正需要的 |
|------|------------|------------|
| 版本管理 | 分支管理 + 三方合并 + 冲突解决 | 简单的"保存历史版本 + 一键回退" |
| 记忆系统 | 三层记忆 + AI 自动提炼 + 衰减公式 | 简单的"记住用户偏好和角色设定" |
| 知识图谱 | 图形化可视化 + AI 自动识别 + 5 万节点支持 | 角色和地点的列表管理 |
| 搜索功能 | 全文搜索 + 向量搜索 + AI 语义搜索 + 融合排序 | 普通的全文搜索 |
| 设计系统 | 三层架构 + 自动同步验证 | 一套统一的样式文件 |

**影响**：按目前规划开发，预计会增加 40-60% 的开发时间，但这些额外功能对第一批用户几乎没有价值。第一批用户需要的是"能用的 AI 写作工具"，不是"功能完备的平台"。

### 问题 3：有些关键环节还只是草案

**通俗解释**：就像汽车设计好了发动机和底盘，但刹车系统还停留在"草案阶段"。

**具体表现**：
- AI 调用的频率限制（防止 API 费用失控）目前只有一份草案，没有实际方案
- 多个模块需要同时操作数据时的安全机制未定义（类似银行转账，扣款和到账必须同时成功或同时失败）
- 上下文组装模块与其他模块的数据接口未定义

## 第三部分：从 CC 能学到什么

CC 的成功不是偶然的。通过深入源码分析，我们提炼出以下可以直接应用于 CN 的核心经验：

### 经验 1：先做管道，再做功能

CC 最核心的设计是一条清晰的**执行管线**：用户输入 → 上下文组装 → AI 调用 → 工具执行 → 结果呈现。所有功能（代码编辑、文件读写、搜索替换等）都是插在这条管线上的"插件"。

**对 CN 的启示**：先建好"用户输入 → 上下文组装 → AI 调用 → 结果应用"这条主管道，然后把润色、续写、查一致性等功能作为插件挂上去。不要先分别做 15 个模块再想办法拼在一起。

### 经验 2：权限和安全是第一天就设计的

CC 从第一行代码就设计了权限系统——每个工具执行前都要检查权限、每次文件修改都有快照备份。不是在功能做完后才"加上安全层"。

**对 CN 的启示**：API Key 安全存储、AI 调用频率限制、文件操作的版本快照——这些应该在第 1 阶段就实现，不是留到后面。

### 经验 3：用户体验比功能数量重要

CC 的核心功能其实只有一个：与 AI 对话并执行代码操作。但它把这一个功能做到了极致——流式输出、实时进度、一键撤销、智能上下文管理。

**对 CN 的启示**：V1 只需要把"AI 辅助写作"这一个核心场景做到极致，而不是同时做知识图谱可视化、三方版本合并、向量语义搜索等高级功能。

### 经验 4：上下文管理是 AI 工具的命脉

CC 花了大量精力在"如何组装最有效的上下文"上。它会自动判断当前任务需要哪些信息（项目结构、相关文件、历史对话等），并在有限的 token 预算内做最优选择。

**对 CN 的启示**：对创作工具来说，最重要的上下文是：当前在写的章节 + 故事大纲 + 角色设定 + 用户的写作风格偏好。把这些组装好，比任何高级功能都更能提升 AI 的写作质量。

## 第四部分：推荐的开发路线

### 总原则

```
先骨架后血肉 → 先核心后扩展 → 先能用后好用 → 先验证后优化
```

### 阶段 1：打地基（预计用时最长，但最关键）

**目标**：把房子的框架搭好，能住人

**要做的事情**：
1. **通信管道**：模块之间怎么说话（IPC 系统——这部分已有优秀设计，可以直接实施）
2. **统一样式**：整个应用长什么样（设计系统的基础部分——一套颜色和字体）
3. **基础布局**：主窗口的框架结构（三栏布局 + 侧边栏 + 状态栏）

**不做的事情**：命令面板、主题切换、快捷键自定义

**完成标准**：打开应用可以看到基本界面框架，模块之间的通信已跑通

### 阶段 2：核心编辑（这是用户最先接触的部分）

**目标**：用户可以在里面写东西

**要做的事情**：
1. **编辑器**：基于 TipTap 2 的富文本编辑器（支持 Markdown 语法、自动保存）
2. **文档管理**：创建/打开/保存文档的基本功能
3. **简单版本管理**：每次保存自动创建版本快照，支持一键回退到某个历史版本

**不做的事情**：分支管理、三方合并、多版本对比、大纲拖拽重排

**完成标准**：用户可以打开应用 → 创建新文档 → 写作 → 保存，并能浏览历史版本

### 阶段 3：接入 AI（这是 CN 的核心价值所在）

**目标**：AI 可以帮助用户写作

**要做的事情**：
1. **AI 管线搭建**：建立从"用户请求"到"AI 响应"的完整执行链路
2. **上下文组装**：自动收集当前章节 + 项目设定作为 AI 的参考信息
3. **核心技能**：实现 3 个最基础的 AI 功能——续写、润色、改写
4. **结果应用**：AI 的输出可以直接预览并一键应用到编辑器中（Inline Diff）
5. **安全防线**：API Key 加密存储 + AI 调用频率限制

**不做的事情**：多候选方案、AI 自动评价（Judge）、自定义技能开发、AI 一致性检查

**完成标准**：用户在编辑器中选中一段文字 → 选择"润色" → AI 返回润色结果 → 用户选择接受/拒绝

### 阶段 4：创作增强（让工具更适合长篇创作）

**目标**：支持长篇创作的管理需求

**要做的事情**：
1. **项目管理**：创建项目 → 项目内管理多个文档 → 项目设定（类型、风格等）
2. **角色/地点管理**：基础的 CRUD 列表（创建、查看、编辑、删除角色/地点信息）
3. **全文搜索**：在整个项目范围内搜索文字
4. **简单记忆**：记住用户的写作偏好和项目级的角色设定，在 AI 调用时自动带上

**不做的事情**：知识图谱可视化、向量语义搜索、RAG、三层记忆架构、AI 自动提炼

**完成标准**：用户可以创建一个小说项目 → 添加角色和地点 → AI 在续写时自动考虑已有的角色设定

### 阶段 5：体验打磨（让好用变成优秀）

**目标**：提升整体使用体验

**要做的事情**：
1. **更多 AI 技能**：一致性检查、对话生成、大纲扩写等
2. **命令面板**：快捷操作入口
3. **多端导出**：支持导出为 Markdown / Word / PDF
4. **主题切换**：亮色/暗色主题
5. **完善的设置面板**：字体、字号、AI 偏好等全部可配置

### 阶段 6（V2）：高级功能

**只有在 V1 稳定运行且有用户反馈支撑时才开始**：
- 知识图谱可视化
- 向量搜索 + RAG
- AI 自动记忆提炼（Distillation Pipeline）
- 版本分支管理
- 自定义 AI 技能
- 多候选方案 + AI 评价系统

## 第五部分：四个关键决策建议

### 决策 1：先补架构再开发

**问题**：目前的 15 个模块设计很详细，但模块之间的协作方案不够。

**建议**：在正式开发前，先补充一份"编排层设计文档"——定义 AI 写作的完整执行链路，让所有模块知道自己在整条链路中的位置和职责。

**通俗比喻**：先画一张整栋楼的管道走线图，再让各个房间的装修团队开工。

### 决策 2：每个模块区分 V1 和 V2

**问题**：很多模块的 V1 设计已经包含了过多功能。

**建议**：对每个模块的设计文档做一次"V1/V2 分离"：

- **V1 标记**：用户能用到的核心功能（happy path + 基本错误处理）
- **V2+ 标记**：高级功能、优化功能、扩展功能

**判断标准**：如果一个功能在前 100 个用户中大概率不会被用到，它就是 V2。

### 决策 3：按依赖顺序而非模块并行开发

**问题**：15 个模块不能同时开发，它们之间有依赖关系。

**建议**：严格按照"阶段 1 → 2 → 3 → 4 → 5"的顺序开发。每个阶段完成后要有一个可以运行的产品，而不是做完 15 个半成品。

**通俗比喻**：先建好一层楼让住户可以搬进来，再盖第二层——而不是同时施工五层楼但每层都没封顶。

### 决策 4：CC 的模式是"学习"而非"移植"

**问题**：CC 和 CN 面向完全不同的用户群体（程序员 vs 创作者），直接移植代码是不可行的。

**建议**：从 CC 学习的是**设计理念和工程实践**，而不是具体代码：
- **学习**：执行管线架构、权限第一理念、上下文管理策略、渐进式复杂度
- **不移植**：CLI 界面、Tool 系统（CC 的 Tool 是文件操作/代码执行，CN 的 Skill 是写作润色/续写）、React+Ink TUI 框架

## 第六部分：最终目标

当 V1 完成时，用户的使用流程应该是这样的：

```
打开 CN → 创建或打开一个创作项目
         → 在编辑器中写作
         → 遇到卡壳时，选中文字，让 AI 帮忙续写/润色/改写
         → AI 理解当前章节内容 + 角色设定 + 用户写作风格
         → 返回高质量的建议
         → 用户一键接受或逐段选择
         → 继续写作
         → 随时可以查看历史版本并回退
         → 完成后导出为 Word 或 PDF
```

这就是 V1 的全部——但每一步都要做得**优秀**。

**衡量成功的标准**不是"有多少功能"，而是：
- 用户愿不愿意每天打开 CN 来写作
- AI 的建议是否真正帮到了创作
- 整个操作流程是否流畅无卡顿
