# 40.4 测试规范（Testing SSOT）

> 本文档是 CreoNow 前端测试的唯一权威规范。完整参考见 [testing-guide.md](../../references/testing-guide.md)。

---

## 一、核心信条

1. **测试行为，不测实现**：断言可观察输出（DOM、状态、返回值），不断言内部调用或 class name
2. **测试是兜底，不是仪式**：每条测试必须回答「防什么回归」，写不出来就不该写
3. **静态约束归 Lint，运行时行为归 Test**：ESLint 能检查的不写 Guard，Guard 检查跨文件约束
4. **否定测试**：对占位 UI、假 Stub、未实现能力，必须写 `should NOT` 测试，防止虚假能力声明
5. **失败信息必须可行动**：测试失败时，错误消息必须能定位问题，不能只说 "expected true, got false"

---

## 二、测试分层

| 层 | 框架 | 测什么 | 文件命名 | 速度 |
|----|------|--------|---------|------|
| **单元** | Vitest | 纯函数、Store action、Hook、映射逻辑 | `*.test.ts(x)` | 快 |
| **组件** | Vitest + RTL | 组件可交互行为（点击、输入、状态切换） | `*.test.tsx` | 快 |
| **集成** | Vitest | 多模块协作（Service + Store + 多组件） | `*.integration.test.ts` | 中 |
| **Guard** | Vitest + FS/AST | 跨文件约束（依赖边界、Token 覆盖率） | `*.guard.test.ts` | 快 |
| **E2E** | Playwright | 关键用户路径（启动→编辑→保存→AI→导出） | `*.e2e.test.ts` | 慢 |

### 怎么选？

- 单函数 / Store / Hook → **单元**
- 组件交互（点击按钮触发行为） → **组件**
- 多模块协作 / IPC 链路 → **集成**
- 跨文件架构约束 → **Guard**
- 关键用户路径 → **E2E**

## 三、前端组件测试规范

### 查询优先级（严格遵守）

```
getByRole > getByLabelText > getByTestId >> getByText
```

### 必须覆盖的行为

每个交互组件至少测试：

| 场景 | 断言 |
|------|------|
| 渲染为正确的 ARIA role | `getByRole('button')` 等 |
| 用户交互触发回调 | `onClick` / `onChange` 被调用 |
| disabled 阻止交互 | `onClick` 不被调用 |
| loading 状态标记 | `data-loading` 属性存在 |
| error 状态可访问 | `aria-invalid` 属性正确 |

### 反模式（违反即打回）

| 反模式 | 为什么有害 | 正确做法 |
|--------|-----------|---------|
| `expect(x).toBeTruthy()` | 不验证对不对 | `toEqual()` / `toHaveTextContent()` 等精确断言 |
| mock 被测对象本身 | 测的是 mock 不是代码 | 只 mock 边界依赖 |
| 只测 happy path | 真实 bug 在 edge/error 路径 | 至少覆盖 happy + error + disabled |
| 字符串匹配源码 | 测的是文本不是行为 | 调用 API → 断言副作用 |
| 无意义测试名 | 失败时不知道什么坏了 | `[前置条件] → [预期行为]` |

## 四、Guard 测试规范

| 规则 | 说明 |
|------|------|
| 必须有 PASS fixture 和 FAIL fixture | 证明 guard 在正确的代码上通过,在违规代码上失败 |
| FAIL fixture 覆盖关键违规模式 | 不能只测最简单 case |
| Guard vs ESLint 边界 | 单文件内规范 → ESLint；跨文件/跨模块约束 → Guard |

---

## 五、测试确定性要求

| 禁止依赖 | 替代方案 |
|---------|---------|
| 真实时间 | `vi.useFakeTimers()` |
| 随机数 | 固定种子 |
| 网络请求 | mock HTTP / IPC |
| LLM API | mock 固定响应 |
| 环境变量 | 测试内显式设置 |

---

## 六、本地验证命令

```bash
# 核心（main + preload）测试
cd apps/desktop && npx vitest run --config vitest.config.core.ts

# 前端（renderer）测试
cd apps/desktop && npx vitest run --config vitest.config.ts

# 类型检查
pnpm typecheck

# Lint
pnpm lint

# Storybook 构建
cd apps/desktop && pnpm storybook:build
```

---

## 七、CI 映射

| CI Job | 本地等价命令 | 门禁说明 |
|--------|-------------|---------|
| `lint-and-typecheck` | `pnpm lint && pnpm typecheck` | 必须全部通过 |
| `test-core` | `pnpm test:unit && pnpm test:integration` | 含 IPC 契约 + Stub 检测 |
| `test-renderer` | `vitest run --config vitest.config.ts` | renderer 测试 |
| `storybook` | `pnpm storybook:build` | 构建产物上传为 Artifact |
| `build` | `pnpm desktop:build` | 依赖前三个 job 全通过 |
