### 这是什么？

AI 倾向于用最简单直接的方式写代码，不会主动使用设计模式或架构约束。久而久之，一个本来设计得不错的项目会**退化成一团意面代码**（所有东西都纠缠在一起）。

打个比方：你规划了一座城市，有清晰的功能分区——住宅区、商业区、工业区。但 AI 像一个不看规划图的建筑工，哪里有空地就在哪里建，最终住宅旁边挨着工厂，商店开在高速公路中间。

### 🔍 具体表现

- **万能文件**：一个文件里又有数据处理、又有 UI 逻辑、又有网络请求——什么都塞在一起
- **忽略分层架构**：不遵守你设计的「主进程/渲染进程」分离、「服务层/数据层」分离
- **直接依赖代替接口**：模块之间直接 import 具体实现，而不是通过接口/抽象层通信
- **God Object（上帝对象）**：一个类或模块承担了太多职责，什么都归它管
- **循环依赖**：A 依赖 B，B 又依赖 A，形成死结
- **业务逻辑散落**：同一个业务规则在 UI 层、服务层、数据层各出现一次，改一处其他地方忘改

### ⚠️ 为什么有害？

1. **这就是你遇到的「重耦合」问题的深层原因**：架构边界不清晰 → 模块之间随意引用 → 越来越紧密耦合
2. **对 CN 的 12 模块路线图是致命的**：如果模块边界被打破，Phase 分期和并行策略就失效了
3. **可测试性崩塌**：耦合在一起的代码无法单独测试
4. **规模不可扩展**：当功能越来越多时，每加一个功能都要改很多地方
