### 这是什么？

这就是你提到的「AI 总给自己留后路」。AI 生成的代码里充满了 try-catch、fallback、默认值、降级逻辑——**代码出了错不会报错，而是悄悄用一个「安全」的默认值糊弄过去**。

打个比方：你让助手帮你打电话订餐厅，助手打不通电话，但不告诉你，而是默默帮你订了一家随便的餐厅。表面上任务完成了，实际上完全不是你要的结果。

### 🔍 具体表现

- **吞掉异常**：`catch(error) { return null }` 或 `catch(error) { return [] }`——出错了就返回空值，不告诉任何人
- **万能 fallback**：任何读取失败都用默认值兜底，导致 UI 上显示的数据可能完全不是真实数据
- **多层降级嵌套**：A 方案失败就试 B，B 失败就试 C，C 失败就返回空——每一层都在吞掉错误信息
- **「安全」的空操作**：函数执行失败后什么也不做，不报错也不记日志，就当没发生过
- **过度的 `|| ''` 和 `?? {}`**：到处用空字符串和空对象做兜底，掩盖了上游传来的 undefined/null 问题

### ⚠️ 为什么有害？

1. **问题被掩盖**：真正的 bug 被 fallback 藏起来了，你以为程序运行正常，其实数据全是错的
2. **调试极难**：出问题时无法定位根因，因为错误在传播过程中被「好心地」吞掉了
3. **虚假的稳定性**：测试全部通过（因为从不报错），但程序行为完全不符合预期
4. **在 CN 项目中尤其危险**：IPC 通道、AI Provider 调用、文件 I/O 这些关键链路如果静默降级，会导致用户数据丢失或 AI 功能静默失效
