### 这是什么？

AI 特别喜欢把代码拆成一个个小函数，美其名曰「模块化」。但很多时候，这些小函数**只被调用一次**，或者做的事情**极其简单**（比如一行代码包一层函数），反而让代码变得更难读。

打个比方：你让人帮你写一篇文章，结果每句话都被拆成一个单独的段落，每个段落还有一个标题。看起来很「结构化」，读起来却支离破碎。

### 🔍 具体表现

- **一行包装函数**：把 `a + b` 包成 `function add(a, b) { return a + b }`，然后到处调用 `add()`
- **Utils/Helpers 文件爆炸**：出现 `utils.ts`、`helpers.ts`、`common.ts` 等垃圾抽屉文件，塞满了零散的小函数
- **过早抽象**：只有一个地方用到某个逻辑，AI 就把它抽成一个「通用」函数，增加了一层不必要的间接性
- **抽象层叠加**：A 调用 B，B 调用 C，C 调用 D——每一层都只是简单转发，真正干活的只有最底层

### ⚠️ 为什么有害？

1. **阅读成本暴增**：要理解一个功能，需要跳转 5-6 个文件，追踪调用链
2. **调试困难**：出了问题不知道该看哪一层
3. **与「重复链路」问题叠加**：多个辅助函数做的事情差不多，但因为命名不同而共存
4. **虚假的「好代码」感**：看起来很整洁，实际上是过度工程化

### ✅ Owner Review 要点

- [ ]  搜索项目中所有 `utils`、`helpers`、`common` 文件——这些通常是重灾区
- [ ]  对只调用一次的函数提出质疑：**「这个函数有必要单独存在吗？」**
- [ ]  要求 Agent 遵循**「三次法则」**：一个逻辑被使用三次以上才值得抽象
- [ ]  「不要创建 utils/helpers 垃圾抽屉文件；不要对只使用一次的逻辑做函数抽取」
